# Execution Quality Rule — 执行质量标准

> 目的：每轮任务执行时确保产出质量，不流于表面。
> 这些规则在每次 Agent 对话中自动生效。

---

## 核心原则（Karpathy 原则 — Trae 版本）

1. **Think Before Coding** — 分析需求 → 声明假设 → 制定计划 → 执行。不确定就问，禁止静默决策
2. **Simplicity First** — 不超范围加功能，不为单用点做抽象，用最少代码解决问题
3. **Surgical Changes** — 只改请求相关的代码，匹配现有风格，清理自己产生的孤儿代码
4. **Goal-Driven Execution** — 先定义成功标准，再执行。多步骤任务给出 brief plan

## TodoWrite 强制规则

### 何时必须使用 TodoWrite

| 场景 | 要求 |
|------|------|
| 任务涉及 ≥3 个独立步骤 | 必须创建 TodoWrite 列表 |
| 任务涉及跨文件修改 | 必须创建 TodoWrite 列表 |
| 任务有依赖关系（先 A 后 B） | 必须创建 TodoWrite 列表 |
| 用户要求"仔细做"/"要评审" | 必须创建 TodoWrite 列表 |

### 使用规范

```
正确做法：
  1. 任务开始 → 创建 todo 列表（status: pending）
  2. 开始某一步 → 标记为 in_progress（同时只一个）
  3. 完成某一步 → 立即标记为 completed
  4. 全部完成 → 执行 guard-rail 自检 → 向用户报告

错误做法：
  × 做完所有事才补 TodoWrite
  × 批量标记多个 completed
  × TodoWrite 和实际执行不一致
  × 留 pending 项不处理就结束
```

## 调研类任务质量标准

| 维度 | 最低标准 |
|------|---------|
| 数据来源 | ≥2 个独立来源，非 AI 内生知识 |
| 对比分析 | 有结构化对比表（方案/工具/框架 vs 备选） |
| 引用证据 | 关键结论有来源链接或文件路径 |
| 结论 | 明确回答用户的原始问题，不含糊 |
| 残余问题 | 列出"还需要了解的"或"未覆盖的" |

---

## 代码类任务质量标准

| 维度 | 最低标准 |
|------|---------|
| 安全 | 无硬编码密钥/Token/路径，输入验证完整 |
| 风格 | 与周边代码风格一致（缩进、命名、注释习惯） |
| 完整性 | 不只是"核心逻辑"，还有边界处理、错误处理 |
| 可运行 | 修改后的代码没有语法错误（显式验证） |
| 副作用 | 不修改无关文件，不引入不需要的依赖 |

## 文档类任务质量标准

| 文档类型 | 最低字数 | 必须包含 |
|---------|---------|---------|
| 调研报告 | ≥500 字 | 背景/现状/对比/结论 |
| 技术方案 | ≥1000 字 | 架构/接口/数据流/异常处理/选型依据 |
| PRD | ≥1500 字 | 目标/用户/场景/功能/验收/非目标 |
| 评审报告 | ≥500 字 | 检查面/finding/结论/建议 |

---

## 上下文管理

当对话轮次多、上下文逐渐增长时：
- **优先传递文件路径**，而非全量内容
- 同类重复信息指向上轮结论即可，不要重复粘贴
- 大段代码修改用 SearchReplace，不要全文重写
