# Guard Rail Rule — 交付前自检（每次任务强制）

> 触发时机：Agent 向用户宣称"完成/好了/已处理"之前。
> 目的：降低 AI 幻觉，防止交付物只有骨架没有实质内容。

---

## 强制执行规则

### Rule 1: 完整性 Self-Audit（不可跳过）

**向用户报告完成之前，必须在回复中执行以下自检，逐条确认：**

```
=== 完整性自检 ===

□ 我是否逐条覆盖了用户的所有问题/需求？
  - 用户提了 X 个问题/要求
  - 我逐一回应了: [列出]

□ 我是否把应该独立成文的内容合并成了摘要？
  - 如果源材料或需求要求详细输出，我的回复是否有实质内容而非一两句话？
  - [ ] 否，每个问题都有充分展开

□ 我是否有"用一两句话带过"替代实质分析？
  - [ ] 否，关键分析都有证据支撑

□ 我的回复是否达到了该任务类型应有的深度？
  - 调研类: ≥ 3 参考来源 + 对比分析
  - 技术方案: 输入/输出/边界/异常完整
  - 代码实现: 安全 + 正确 + 风格一致
  - [ ] 是

□ 我是否正确加载了相关规范契约?
  - [ ] CLAUDE.md 已读取
  - [ ] 相关 .claude/contracts/ 已参考

=== 成熟度自检 ===

□ 我是否跳过了应走的步骤？
  - [ ] 否，该调研的调研了，该对比的对比了

□ 我是否先写完再回头解释"我做了啥"？
  - [ ] 否，是在分析过程中自然产出的

□ 我对用户的追问是否给的是"再来点"式敷衍？
  - [ ] 否，每个追问都有新增实质性信息
```

**硬规则**：
- 任何标为"否"的项 → 必须修复后再报告完成
- 不得跳过自检直接说"任务完成"
- 自检结果必须显式出现在输出中

### Rule 2: 内容完整性对标（源材料适配场景）

当任务涉及从源材料提取/适配内容时，必须在交付物末尾附对标表：

| 源材料条目标识 | 是否已适配 | 适配位置 | 未适配原因 |
|--------------|-----------|---------|-----------|
| ... | ✅ | ... | - |

**禁止**: "已覆盖"含糊带过，必须有具体适配位置。

### Rule 3: TodoWrite 完整性

如果任务使用了 TodoWrite:
- [ ] 所有创建的任务项是否都已标记为 completed？
- [ ] 是否有新的待办项未追踪？

## 反幻觉锚定规则

以下行为**绝对禁止**：

1. **编造不存在的文件路径** — 写文件前必须确认目录存在
2. **声称读了不存在的文件** — 除非 Read 工具返回了真实内容
3. **虚构 API/库功能** — 不假称某库"支持 X"除非已验证
4. **制造虚假的搜索/引用结果** — 不说"网上都说..."除非真的搜了
5. **把"看起来合理"当事实** — 不确定就说"需要验证"

违反以上任何一条 → 必须立即纠正并如实告知用户。
