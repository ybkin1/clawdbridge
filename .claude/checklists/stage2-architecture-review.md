# Stage 2: 架构拆解评审清单

## 完整性

- [ ] 是否覆盖所有功能需求
- [ ] 系统上下文是否清晰（参与者/外部依赖/边界/非目标）
- [ ] 能力图是否完整（业务能力/用例/角色/成功动作）
- [ ] 模块/组件图是否完整（模块/组件/目录边界/所有权）

## 选型合理性

- [ ] 语言/框架/数据库选型是否有依据
- [ ] 是否有备选方案
- [ ] 选型理由是否充分

## 模块交互

- [ ] 组件间通信方式是否明确
- [ ] 数据流转路径是否清晰
- [ ] 同步/异步策略是否合理
- [ ] 依赖方向是否单向（无循环依赖）

## 可维护性

- [ ] 目录结构规范是否明确
- [ ] 分层原则是否清晰
- [ ] 扩展点是否预留
- [ ] 日志/审计/诊断/回滚入口是否设计

## 追溯性

- [ ] 每个 architecture node 是否有明确职责
- [ ] 每个 node 是否有验证路径
- [ ] 每个 node 是否有回滚/隔离路径
- [ ] 是否可追溯到需求/intent

## 乐高拆解（Rule 7）

- [ ] 是否产出 `lego-decomposition-tree.yaml`
- [ ] 是否从 L0（成品）拆到 L4（代码块）
- [ ] 每个层级的输入/输出/拼装方式是否定义
- [ ] 拆解是否达到"可独立制造"的停止条件
- [ ] 如果 L3 仍复杂到需要 3+ 个代码块，是否提升为 L2
