# 跨层评审检查单 (Cross-Level Review Checklist)

> 版本: v1.0 | 日期: 2026-05-07
> 来源: `recursive-decomposition-spec.md` §2.5
> 适用场景: L3 设计完成、L4 设计完成、L5 代码完成时的跨层对齐评审

---

## 使用说明

本检查单由 **Cross-Level Reviewer Agent** 执行。评审时只需加载：
1. 被评审的设计文档（L3/L4/代码）
2. 顶层文档路径（PRD 或 L2 架构）
3. tree.json 中的锚链数据

**不加载**：全量树、其他分支的设计细节。

---

## 1. ORPHAN 检查（需求追溯）

| # | 检查项 | 通过标准 |
|---|--------|---------|
| 1.1 | 该组件/单元能否追溯到 L1 PRD 中的具体功能需求？ | 能定位到 PRD 中的具体功能 ID 或描述 |
| 1.2 | 该组件/单元是否在 L2 架构中有对应的模块归属？ | 能定位到 L2 架构中的父模块 |
| 1.3 | 该组件的职责是否在锚链的 `l2_context` / `l3_context` 覆盖范围内？ | 是，没有超出父层定义的范围 |

**如果任何一项为否 → 标注 ORPHAN，需解释为什么存在。**

---

## 2. VIOLATION 检查（黄金约束遵守）

| # | 检查项 | 验证方式 |
|---|--------|---------|
| 2.1 | 是否违反了锚链中列出的任何一条 `golden_constraints`？ | 逐条对照锚链中的约束编号 |
| 2.2 | 设计是否隐含了"替代现有系统"（ITR/ERP/CRM/WMS）？ | 检查是否有创建替代系统的意图 |
| 2.3 | AI 功能是否优先使用本地 vLLM，而非默认走在线 API？ | 检查模型调用路径 |
| 2.4 | 数据是否指向外部云服务（AWS/GCP/阿里云等）？ | 检查数据库、缓存、消息队列的连接配置 |
| 2.5 | 是否针对 10 种用户角色做了差异化设计，而非"统一界面"？ | 检查是否有角色相关的权限/UI/流程差异 |

**如果任何一项为是 → 标注 VIOLATION，必须修复，不允许 Conditional Pass。**

---

## 3. ASSUMPTION_DRIFT 检查（假设一致性）

| # | 检查项 | 对照来源 |
|---|--------|---------|
| 3.1 | 该组件的设计假设（用户行为、数据量、并发量等）是否与 PRD 中的假设一致？ | PRD §非功能需求 |
| 3.2 | 该组件是否引入了 PRD 中未提及的新假设？ | PRD 全文 |
| 3.3 | 如果假设有变化，是否合理？（如 PRD 说 50 并发，这里设计为 500） | 有数据或业务理由支撑 |

**如果有不一致且无合理理由 → 标注 ASSUMPTION_DRIFT，回到 L1 确认。**

---

## 4. TECH_DRIFT 检查（技术选型一致性）

| # | 检查项 | 对照来源 |
|---|--------|---------|
| 4.1 | 该组件的技术选型（语言、框架、协议）是否与 L2 架构的技术栈一致？ | L2 架构文档技术栈章节 |
| 4.2 | 是否引入了 L2 架构中未提及的新依赖？ | L2 架构文档依赖清单 |
| 4.3 | 如果技术选型有偏离，是否写了 ADR 说明理由？ | 对应 ADR 文件 |

**如果有偏离但无 ADR → 标注 TECH_DRIFT，要求补充 ADR。**

---

## 5. 锚链完整性检查

| # | 检查项 | 通过标准 |
|---|--------|---------|
| 5.1 | 设计文档头部是否包含锚链摘要（溯源链 + 黄金约束编号）？ | 是 |
| 5.2 | 锚链中的 `golden_constraints` 编号是否有效（不超过 root 锚中定义的约束数）？ | 是 |
| 5.3 | 锚链是否被不当修改（上层内容被删除或更改）？ | 对照 tree.json 中的父锚，无篡改 |

**如果任何一项为否 → 标注 ANCHOR_INVALID，必须修正。**

---

## 评审结果判定

| 结果 | 判定条件 | 动作 |
|------|---------|------|
| **Pass** | 无 ORPHAN、无 VIOLATION、无 ASSUMPTION_DRIFT、无 TECH_DRIFT（或 TECH_DRIFT 有有效 ADR） | 评审通过 |
| **Conditional Pass** | 有 ASSUMPTION_DRIFT 但已确认合理 / 有 TECH_DRIFT 且 ADR 完整 | 通过，记录遗留项 |
| **Fail** | 任何 ORPHAN 或 VIOLATION | 必须修复，进入修复轮 |

---

## Fix Brief 模板（发现偏离时使用）

```markdown
## Fix Brief — Cross-Level
- **偏离类型**: ORPHAN / VIOLATION / ASSUMPTION_DRIFT / TECH_DRIFT / ANCHOR_INVALID
- **偏离位置**: {文件路径} §{章节}
- **顶层对照**: {PRD/架构文档} §{章节}
- **具体问题**: {描述偏离}
- **修复要求**: {具体修复指令}
- **优先级**: P0(VIOLATION/ANCHOR_INVALID) / P1(ORPHAN) / P2(DRIFT)
```
