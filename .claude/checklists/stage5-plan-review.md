# Stage 5: 开发计划 评审清单

## 来源
Rule 2: Mandatory Deep Iterative Review — Covered outputs（开发计划扩展产物）

## 评审维度

### 可执行性
- [ ] 每个任务是否可追溯到方案详设中的具体章节
- [ ] 架构拆解中的每个最小积木块是否有对应的开发任务
- [ ] 任务粒度是否合理（单个任务≤2天工作量）
- [ ] 任务依赖关系是否明确
- [ ] 关键路径是否标注

### 风险覆盖
- [ ] 技术风险点是否识别并有应对方案
- [ ] 外部依赖风险是否识别（第三方API/审批/资源）
- [ ] 是否有缓冲时间/回退方案

### 里程碑
- [ ] 里程碑节点是否可交付（不是"进行中"）
- [ ] 每个里程碑是否有明确的验收标准
- [ ] 里程碑之间的依赖关系是否合理

### 资源匹配
- [ ] 任务复杂度与人员/工具能力是否匹配
- [ ] 是否有并行执行的机会
- [ ] 测试/联调时间是否预留

### 设计-计划双向追溯（新增）
- [ ] 每个开发包是否有 @ref 指向方案详设 §14 具体 STEP（精确到 STEP N）
- [ ] 方案详设 §14 每个 STEP 是否有 ≥1 个开发包覆盖（双向引用矩阵检查）
- [ ] 未覆盖的 STEP 是否标记为 `deferred` 或 `not_applicable` 并有理由
- [ ] 未覆盖的 STEP / orphan 包是否在 exceptions 列表中声明且有 `approved_by` 审批人
- [ ] 开发包中是否有 orphan 包（无 @ref 来源）？如有，是否标记为 `glue_code`

### 包级约束（新增）
- [ ] 每个包是否声明行数预估？是否 ≤50 行或声明 oversized_justified？
- [ ] 超出 50 行的包是否声明 `oversized_reason` 且值为 4 类允许值之一（纯数据结构/原子逻辑/同模式批量/强耦合胶水）？
- [ ] 每个包是否有 @packet/@input/@output/@contract/@ref/@test 注释模板？
- [ ] 包的验收标准是否是「可自动验证」的一句话（非模糊描述如"功能正常"）
- [ ] 包的依赖关系是否无环？关键路径是否标注？

### 合入计划（新增）
- [ ] 是否有单元级合入计划（每包完成后合入哪个分支）？
- [ ] 是否有线程安全声明（哪些包可并行、哪些必须串行）？
- [ ] 拼装顺序是否遵循 contract first → implementation → entry last？
- [ ] 每个合入点是否有回滚策略？

### verdict
- Pass: 无Critical，Major已修复或有修复计划，Minor≤3
- Conditional Pass: 无Critical，Major已接受并有缓解方案
- Fail: 任何Critical未解决
