# Stage 6: 编码 评审清单

## 来源
Rule 2: Mandatory Deep Iterative Review — Covered outputs

## 评审维度

### 场景走读（第一优先级，详见 scenario-traceability-spec.md）
- [ ] 是否已建立追溯矩阵：需求来源 → 用户场景 → 场景路径 → 代码位置 → 测试用例
- [ ] 是否对每个 P0/P1 场景按步骤逐行走读了代码
- [ ] 走读时是否代入用户角色操作（不是"代码写得好不好"，是"用户这样操作能不能跑通"）
- [ ] 每个场景的完整路径是否有对应代码实现（断点/卡死/数据丢失/状态不一致）
- [ ] 发现的问题是否标注：场景、代码位置、问题类型、影响的功能、复现路径
- [ ] 需求来源未覆盖的隐式场景是否已补充（断网重连、首次启动、后台杀死、并发操作）
- [ ] 是否满足 scenario-traceability-spec.md §5.3 最低场景目录要求（P0 happy path + 网络异常 + 异常输入 + 平台/安全特定场景）
- [ ] 是否触发 scenario-traceability-spec.md §7 任一浅层走读硬失败条件
- [ ] 涉及认证/加密/PII 时是否已增加 scenario-traceability-spec.md §5.5 安全场景
- [ ] 追溯矩阵中引用的代码行号是否有效（行号存在且与变更范围可对应）

### 正确性
- [ ] 代码是否严格遵循方案详设中的接口契约和数据模型
- [ ] 核心算法逻辑是否与方案详设中的伪代码/流程一致
- [ ] 状态机实现是否完整覆盖方案详设中的所有状态和流转条件

### 安全性
- [ ] 输入验证：所有外部输入是否经过类型检查和边界验证
- [ ] 认证/授权：敏感操作是否有正确的权限校验
- [ ] 数据脱敏：日志/响应中是否泄露敏感信息
- [ ] 常见漏洞：SQL注入、XSS、CSRF、路径遍历等

### 可维护性
- [ ] 每个最小积木块（函数/组件/API）是否有对应代码实现
- [ ] 模块/组件间调用是否遵循架构拆解中的分层原则
- [ ] 是否遵循目录结构规范
- [ ] 变量/函数命名是否一致
- [ ] 代码是否遵循 Karpathy Simplicity 原则（无多余特性、无过度抽象）

### 性能
- [ ] 是否存在N+1查询
- [ ] 是否存在不必要的全表扫描/全列表遍历
- [ ] 缓存使用是否合理
- [ ] 异步/并发策略是否正确

### 测试配套
- [ ] 核心函数是否有对应单元测试
- [ ] 模块间交互是否有对应集成测试
- [ ] 核心用户场景是否有对应端到端测试

### 代码溯源（新增）
- [ ] 每个新增代码文件头部是否有 `@packet <packet_id>` 注释？
- [ ] 每个 export 函数/类是否在 Dev Plan 中有对应包？
- [ ] git diff 中是否存在 orphan 代码（无对应 Design §14 STEP + 无对应 Dev Plan 包）？
- [ ] 代码中的异常处理路径是否能在 Design §14 异常传播链中找到来源？
- [ ] 代码中的状态赋值是否能在 Design 状态变更追踪中找到来源？

### 包级验收（新增）
- [ ] 每个包的 @test 验收标准是否已通过？
- [ ] 包级单元测试是否覆盖正常路径 + 主要异常路径？
- [ ] 包是否符合 @contract 定义的输入/输出签名？
- [ ] oversized 包是否声明 `oversized_review` 字段且有评审人批准？

### 拼装验证（新增）
- [ ] L4→L3 拼装后是否通过集成测试？
- [ ] L3→L2 拼装后是否通过模块测试？
- [ ] 拼装后的接口类型是否对齐（TypeScript 编译通过 / Python 类型检查通过）？
- [ ] 拼装后的模块间调用是否符合 §14.0 依赖注入总表？

### code-review-graph 图谱分析
- [ ] 是否已执行 `detect_changes_tool` 获取风险评分和优先级排序
- [ ] 是否已执行 `get_affected_flows_tool` 识别受影响的用户流程
- [ ] 高风险项（risk score 前3）是否逐一审查
- [ ] 变更影响的执行流是否有对应测试覆盖
- [ ] 桥接节点 / 枢纽节点变更是否评估回归风险
- [ ] 知识图谱中的未测试热点是否标记

### verdict（Stage 6 整体门控）
- Pass: 无Critical，Major已修复或有修复计划，Minor≤3；所有 packet 已完成包级评审且拼装验证通过
- Conditional Pass: 无Critical，Major已接受并有缓解方案；允许存在未完成的拼装层级但已记录修复计划
- Fail: 任何Critical未解决；或存在未通过包级评审/拼装评审且超过 3 轮的 packet

> **与 packet-level / assembly-level 的区别**：单个 packet 的包级评审失败只阻塞该 packet 向上拼装，不直接导致 Stage 6 Fail。Stage 6 Fail 仅在以下情况触发：(a) 拼装覆盖率 < 阈值，(b) 最终集成测试失败，(c) 某 packet Critical 未修复且超过 3 轮。
