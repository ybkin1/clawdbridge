# Stage 6: 代码评审清单

## 安全

- [ ] 无注入漏洞（SQL/XSS/命令注入）
- [ ] 无硬编码密钥/凭证
- [ ] 输入验证是否到位
- [ ] 权限/鉴权是否正确实现
- [ ] 敏感数据是否加密/脱敏

## 正确性

- [ ] 逻辑是否与 PRD/设计一致
- [ ] 边界条件是否处理
- [ ] 异常路径是否处理
- [ ] 幂等性/重试是否实现（如需要）
- [ ] 竞态条件是否考虑

## 性能

- [ ] 是否有 N+1 查询
- [ ] 是否有不必要的循环/递归
- [ ] 是否有内存泄漏风险
- [ ] 大输入是否有性能退化

## 代码风格

- [ ] 是否匹配项目现有风格
- [ ] 命名是否清晰
- [ ] 是否有不必要注释
- [ ] 是否有死代码（来自本次改动的）

## Karpathy 原则

- [ ] 是否过度抽象（为单用点做抽象）
- [ ] 是否有超范围功能（speculative feature）
- [ ] 是否只改了请求相关的代码
- [ ] 是否有不必要的错误处理（处理不可能的场景）
- [ ] 复杂度是否与任务匹配

## Work Packet 合规

- [ ] 是否符合 work packet 定义的 write_scope
- [ ] 是否符合 12 层拆包要求
- [ ] 是否可追溯到 architecture node
- [ ] 是否遵守拆包顺序

## 乐高拼装合规（Rule 7）

- [ ] 当前代码是否对应 lego-decomposition-tree.yaml 中的一个 L4 代码块
- [ ] 代码实现是否与 Stage 4 方案详设中对应单元/代码块的设计一致
- [ ] 如果拼装到 L3+ 层级，是否有对应的单元测试验证拼装正确性
- [ ] 低层级拼装是否已验证通过后才进行高层级拼装
- [ ] 是否有未恢复的 Agent 异常（挂死/产出不足/质量不合格）
