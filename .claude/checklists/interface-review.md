# 接口评审检查单 (Interface Review Checklist)

> 版本: v1.0 | 日期: 2026-05-07
> 来源: `recursive-decomposition-spec.md` §5.4 评审层级匹配
> 适用场景: L2/L3 间接口契约创建或变更时

---

## 使用说明

本检查单由 **Reviewer Agent** 或 **Cross-Level Reviewer Agent** 执行。评审时加载：
1. 被评审的接口契约文档
2. 提供方和消费方的设计文档
3. tree.json 中的接口定义

---

## 1. 契约完整性

| # | 检查项 | 通过标准 |
|---|--------|---------|
| 1.1 | 接口类型是否明确？（REST / MCP Tool / NATS Event / 函数调用） | 类型字段非空且有效 |
| 1.2 | 端点/工具名是否完整列出？ | 所有端点都有定义 |
| 1.3 | 请求格式是否使用 Pydantic/JSON Schema 定义？ | 有明确的 schema |
| 1.4 | 响应格式是否使用 Pydantic/JSON Schema 定义？ | 有明确的 schema |
| 1.5 | 提供方和消费方是否明确标注？ | provider 和 consumer 非空 |

---

## 2. 双向一致性

| # | 检查项 | 验证方式 |
|---|--------|---------|
| 2.1 | 提供方定义的输出是否与消费方期望的输入匹配？ | 对照双方设计文档 |
| 2.2 | 消费方是否依赖了提供方未声明的接口？ | 检查 consumer imports |
| 2.3 | 多个消费方对同一接口的使用是否一致？ | 检查各消费方设计文档 |

---

## 3. 约束与可靠性

| # | 检查项 | 通过标准 |
|---|--------|---------|
| 3.1 | 超时是否定义？ | 超时值 > 0 且合理 |
| 3.2 | 重试策略是否定义？ | 有重试次数和退避策略 |
| 3.3 | 错误码是否覆盖所有异常路径？ | 至少覆盖 4xx/5xx |
| 3.4 | 幂等性是否声明？ | 是/否 明确标注 |

---

## 4. 版本兼容性

| # | 检查项 | 通过标准 |
|---|--------|---------|
| 4.1 | 接口是否有明确版本号？ | v1.0 / v2.0 等 |
| 4.2 | 破坏性变更流程是否定义？ | 有迁移/适配策略 |
| 4.3 | 旧版本是否标注了废弃时间？ | deprecated 标记 + 时间 |

---

## 5. 黄金约束一致性（Cross-Level）

| # | 检查项 | 对照来源 |
|---|--------|---------|
| 5.1 | 接口是否违反 golden_constraints？ | tree.json root anchor |
| 5.2 | 接口是否引入外部云服务依赖（违反数据本地化）？ | 检查连接配置 |
| 5.3 | 接口设计是否隐含"替代现有系统"？ | 对照 Out of Scope 清单 |

---

## 评审结果判定

| 结果 | 判定条件 | 动作 |
|------|---------|------|
| **Pass** | 所有检查项通过 | 接口契约确认 |
| **Conditional Pass** | Minor 问题 ≤2，有修复计划 | 确认，限期修复 |
| **Fail** | 任何 Critical 项未通过 | 必须修复，进入修复轮 |

---

## Fix Brief 模板（发现偏离时使用）

```markdown
## Fix Brief — Interface
- **偏离类型**: INCOMPLETE / INCONSISTENT / MISSING_VERSION / GOLDEN_VIOLATION
- **偏离位置**: {接口契约文件路径} §{章节}
- **提供方**: {provider-id}
- **消费方**: {consumer-ids}
- **具体问题**: {描述}
- **修复要求**: {具体修复指令}
- **优先级**: P0(双向不一致/黄金约束违反) / P1(约束缺失) / P2(版本缺失)
```
