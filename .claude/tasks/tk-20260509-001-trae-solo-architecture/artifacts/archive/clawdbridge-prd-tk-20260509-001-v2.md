## 1. 产品概述

### 1.1 一句话描述

**ClawdBridge Mobile** 是一个纯对话式手机客户端，通过局域网/互联网连接桌面端的 Bridge Server（基于 clwd-on-desk 扩展），用自然语言驱动桌面端 **Claude Code CLI** 执行编码、文件操作、Git 工作流等任务，手机上只看结果、批审批，不写一行代码。

### 1.2 产品定位

```
                    Trae Solo              ClawdBridge Mobile
                    ─────────              ──────────────────
AI 引擎             豆包/DeepSeek           Claude Code (原生 CLI)
Agent 架构          双智能体( Builder+Coder) Builder(手机) + Coder(Claude Code)
执行端              云端或桌面 Agent         只有真实桌面端
数据同步            State Sync              State Sync (借鉴)
审批机制            App 内弹窗              推送卡片 → 一键审批
多端               Web+桌面+手机            手机 ↔ 桌面（+Web 可选）
```

### 1.3 核心价值主张

| 痛点 | 解决方案 |
|------|---------|
| Trae Solo 不支持 Claude Code | ✅ 原生驱动 Claude Code CLI |
| 桌面 Claude Code 审批需逐个窗口点 | ✅ 手机统一审批，一个卡片处理 |
| 离开电脑任务就中断 | ✅ 手机远程下发，桌面持续执行 |
| 手机打字效率低 | ✅ 语音输入，自然语言下指令 |
| 多设备切换丢上下文 | ✅ State Sync 对话历史全量同步 |

## 2. 市场背景与产品目标

### 2.1 时代背景

- Claude Code 已成为专业开发者的首选 AI 编程工具，但**绑定桌面终端**
- Trae Solo 证明了"纯对话式 AI 编程"可在手机端跑通，三端协同模式验证
- 现有移动端编程工具（Cursor/GitHub Copilot）均无 Claude Code 集成
- 创业者需要一款"手机 Claude Code 遥控器"

### 2.2 产品目标

**Phase 1（MVP）**：
- 手机可通过局域网连接桌面 Bridge Server
- 支持纯文本对话 + Claude Code 指令下发
- 审批推送到手机端，一键同意/拒绝
- 对话历史本地 + Bridge Server 双写同步
- WebSocket 前台实时通知（审批卡片 / 执行完成 / 错误）

**Phase 2**：
- 语音输入与播报
- 互联网远程连接（通过轻量中继服务器）
- Web 端 PWA
- GitHub 集成（任务产物自动同步）

**Phase 3**：
- 多桌面端接入与任务调度
- 定时任务与预设计划
- 文件预览（图片/PDF/Markdown）
- 后台推送通知（FCM/APNs，App 在后台时接收）

## 3. 目标用户与使用场景

### 3.1 用户画像

| 画像 | 特征 | 核心诉求 |
|------|------|---------|
| **离开桌面的开发者** | 通勤/外出时需跟进任务 | 远程看进度、审批、微调指令 |
| **不想坐电脑前的创业者** | 纯对话式开发，不碰代码 | 手机说完需求，等桌面交付 |
| **多任务并行者** | 同时跑多个项目/仓库 | 手机统一监控所有任务 |
| **灵感驱动型创作者** | 碎片时间产生想法 | 语音记录 → 直接转为可执行任务 |

### 3.2 核心场景

| 场景编号 | 场景描述 | 预期行为 |
|---------|---------|---------|
| **S1 — 通勤下发任务** | 地铁上想到一个 Bug 修复方案 | 打开 App → 语音说出修复思路 → Claude Code 在桌面自动执行 |
| **S2 — 审批免打断** | 桌面运行中遇到文件写入审批 | 手机收到推送卡片 → 查看操作详情 → 一键允许 |
| **S3 — 外出查看进度** | 出差时想了解上午任务的进展 | 打开 App → 任务列表显示各任务状态 → 进入对话查看执行日志 |
| **S4 — 碎片时间优化设计** | 等餐时想调整方案 | 手机发消息："把刚才生成的 API 接口加一个分页参数" → 桌面端接续执行 |
| **S5 — 多项目监控** | 4 个项目同时有任务在跑 | 任务列表分组 → 实时显示各项目状态 → 快速切换对话 |

## 5. 非功能需求

| 指标 | 目标值 |
|------|--------|
| 消息发送到 Bridge Server 接收延迟 | < 200ms（局域网） |
| 审批卡片弹出响应时间 | < 500ms |
| 对话列表加载（100 条对话） | < 1s |
| App 冷启动到可交互 | < 3s |
| Claude Code 流式响应首 Token | 取决于模型，不做额外延迟 |
| 内存占用（空闲状态） | < 80MB |
| 内存占用（活动对话） | < 150MB |

### 5.2 可靠性

| 指标 | 目标值 |
|------|--------|
| WebSocket 断线自动重连 | 最多 5 次，间隔 2s 递增 |
| 审批消息不丢失 | At-least-once 投递 + 去重 |
| 本地数据库崩溃 | SQLite WAL 模式自动恢复 |
| 服务可用性 | 99%（不依赖外部云端的 MVP 阶段） |

### 6.2 技术选型

| 层 | 选型 | 理由 |
|----|------|------|
| **移动端框架** | React Native 0.76+ + Expo | 一套代码双端运行，热更新快，生态丰富 |
| **状态管理** | Zustand | 轻量、TS 友好、适合实时状态 |
| **本地存储** | expo-sqlite (SQLite) | 对话历史离线缓存 |
| **实时通信** | WebSocket (react-native-websocket) | 全双工低延迟 |
| **语音识别** | expo-speech-recognition | 系统级 ASR |
| **推送** | Firebase Cloud Messaging / APNs | 标准推送通道 |
| **认证** | JWT + OAuth 2.0 (expo-auth-session) | 标准方案，可复用 Trae Solo 参考架构 |
| **Bridge Server** | Node.js + Express + ws | 现有 clwd-on-desk 技术栈 |

### 6.3 关键模块设计

#### 6.3.1 消息协议

所有手机 ↔ Bridge 消息统一 JSON 协议：

```json
{
  "id": "msg_uuid",
  "type": "user_message" | "assistant_stream" | "approval_request" | "approval_response" | "tool_status" | "session_state",
  "sessionId": "session_uuid",
  "timestamp": 1703664000000,
  "payload": {
    "content": "...",
    "metadata": {}
  }
}
```

#### 6.3.2 审批流转

```
Claude Code 触发 PermissionRequest
        │
        ▼
Bridge Server Hook 拦截
        │
        ▼
生成 approval_request 消息 → WebSocket → 手机端
        │                                      │
        │                              弹出审批卡片
        │                                      │
        │                              用户点击 允许/拒绝
        │                                      │
        ▼                                      │
  ←─ approval_response 消息                     │
        │
        ▼
Bridge 将决定注入 Claude Code (stdin 或 Hook 回调)
        │
        ▼
Claude Code 继续执行或中止
```

#### 6.3.3 State Sync 模型

借鉴 Trae Solo 的四层同步模型，简化为三层：

```
Layer 2: 会话上下文同步
  ├─ 对话消息全量同步（消息 ID 去重）
  ├─ 会话元数据（标题/创建时间/关联设备）
  └─ 执行状态（running/idle/error）

Layer 1: 审批状态同步
  ├─ 待审批请求队列
  ├─ 审批结果
  └─ 审批超时状态

Layer 0: 设备状态同步
  ├─ Bridge Server 在线/离线心跳
  └─ 配对设备列表
```

## 8. 数据流与接口

### 8.1 连接建立流程

```
App 启动
  ↓
加载本地缓存的会话列表（SQLite）
  ↓
尝试 WebSocket 连接 Bridge Server
  ├─ 连接成功 → 发送 client_connect 消息（含 device_id + token）
  │             → 接收 session_sync 全量同步
  │             → 进入实时通信模式
  └─ 连接失败 → 显示离线状态
                → 使用本地缓存数据
                → 后台定时重连（间隔 2s/5s/15s/30s/60s）
```

### 8.2 核心 API（Bridge Server 端）

| 方法 | 路径 | 说明 |
|------|------|------|
| `POST` | `/api/auth/login` | 手机号/邮箱登录 |
| `POST` | `/api/auth/oauth` | OAuth 回调 |
| `POST` | `/api/auth/refresh` | Token 续期 |
| `GET` | `/api/devices` | 获取已配对设备列表 |
| `POST` | `/api/devices/pair` | 发起设备配对 |
| `DELETE` | `/api/devices/:id` | 解绑设备 |
| `GET` | `/api/sessions` | 获取会话列表 |
| `POST` | `/api/sessions` | 创建新会话 |
| `GET` | `/api/sessions/:id/messages` | 获取会话消息历史 |
| `WS` | `/ws` | 实时通信端点 |

### 8.3 WebSocket 消息类型

| type | 方向 | 说明 |
|------|------|------|
| `client_connect` | 手机 → Bridge | 连接握手 |
| `user_message` | 手机 → Bridge | 用户输入 |
| `assistant_stream` | Bridge → 手机 | Claude Code 流式响应 |
| `tool_invocation` | Bridge → 手机 | 工具调用开始 |
| `tool_result` | Bridge → 手机 | 工具调用结果 |
| `approval_request` | Bridge → 手机 | 审批请求 |
| `approval_response` | 手机 → Bridge | 审批决定 |
| `session_sync` | Bridge → 手机 | 全量会话同步 |
| `session_state` | Bridge → 手机 | 会话状态变更 |
| `ping` / `pong` | 双向 | 心跳保活（30s 间隔） |

## 9. 身份认证与安全设计

### 9.1 认证架构（借鉴 Trae Solo）

```
注册/登录
  ├─ 手机号 + 验证码
  └─ GitHub OAuth 2.0
        ↓
Bridge Server 验证身份
        ↓
签发 JWT Token { user_id, device_id, exp, iat }
        ↓
手机端安全存储 Token (expo-secure-store / Keychain / Keystore)
        ↓
每次请求携带 Authorization: Bearer <token>
WebSocket 连接时 query 参数传 token
```

### 9.2 安全原则

1. **API Key 不落手机**：Claude Code 的 API Key 存储在桌面 Bridge Server 环境变量中，手机端不接触
2. **最小权限**：桌面端仅开放授权目录，手机不能越权访问
3. **审批强制**：工作区外的文件操作、sudo 命令必须手机审批
4. **传输加密**：局域网内 WSS，远程通过中继 TLS
5. **Token 刷新**：短期 access_token（15 min）+ 长期 refresh_token（7 天）

## 13. 风险与依赖

### 13.1 技术风险

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
