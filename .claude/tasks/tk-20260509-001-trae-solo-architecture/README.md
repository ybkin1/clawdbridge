## 一、整体产品生态定位

```

│  - 多人共享 AI 开发成果                                │
│  - 第三方大模型 API 接入                               │
├─────────────────────────────────────────────────────┤

### 2.2 五大核心组件链

Trae Solo 内部基于"指令输入 → 意图识别 → 方案生成 → 任务执行 → 结果反馈"完整闭环：

| 组件 | 功能 | 技术要点 |
|------|------|---------|
| **自然语言指令解析器** | 将中文自然语言转为结构化任务 | 中文语境优化的语义理解，口语化输入 |
| **意图识别引擎** | 区分 Code/MTC 模式，识别核心意图 | 自动判断"写代码"还是"写文档" |
| **方案生成器** | 拆解任务为可执行步骤 | Context Engineering 动态上下文构建 |
| **工具链执行引擎** | 文件编辑、Shell 命令、代码运行、测试 | MCP 协议封装工具能力 |
| **结果反馈系统** | 流式输出执行进度，产物预览 | WebSocket 实时推送，卡片式 UI |

### 2.3 双智能体协作机制

```
用户需求
    ↓
┌─────────────────┐
│  SOLO Builder   │  ← 规划型智能体
│  - 理解需求     │
│  - 拆解任务     │
│  - 调用工具     │
│  - 查文档/画图  │
└────────┬────────┘
         ↓ 任务拆解结果
┌─────────────────┐
│  SOLO Coder     │  ← 执行型智能体
│  - 代码实现     │
│  - 测试修复     │
│  - 调试自愈     │
│  - 部署打包     │
└─────────────────┘
         ↓
     成果交付
```

这跟你的需求高度匹配——你不需要 IDE 编辑器能力，只需要 Builder 拆解任务 → 桌面端 Coder 执行。

### 2.4 Code vs MTC 双模式

| 模式 | 定位 | 能力 |
|------|------|------|
| **Code** | 专业开发场景 | 代码编写、重构、Bug修复、Git工作流 |
| **MTC** | 通用办公场景 | 文档处理、PPT生成、数据挖掘、文案创作 |

来源：[Trae Solo 官方文档](https://docs.trae.ai/solo/what-is-trae-solo)

## 三、身份识别与认证体系

### 3.1 账号注册方式

```
                 ┌─ 手机号注册（国内版，符合实名制要求）
                 │
用户注册 ────────┼─ 第三方登录 ── Google 账号
                 │              ├─ GitHub 账号
                 │              └─ 字节系账号关联
                 │
                 └─ 国际版 Pro 账号（付费）
```

来源：[Trae 用户协议](https://www.trae.com.cn/terms-of-service/cn) §4.1

### 3.2 认证令牌体系（推断）

基于隐私政策和 API 协议推断的认证链路：

```
┌──────────────────────────────────────────────────────────────┐
│  1. 用户输入手机号 → 获取短信验证码                              │
│  2. 或选择第三方登录 → OAuth 2.0 回调获取 access_token          │
│  3. 后端签发 JWT Token（含 user_id + 设备信息 + 过期时间）       │
│  4. 客户端存储 JWT → 每次 API 请求携带 Authorization Header     │
│  5. WebSocket 连接时携带 token 鉴权                            │
└──────────────────────────────────────────────────────────────┘
```

关键特征：
- **TRAE 与 SOLO 账号通用**：同一账号可在 IDE 和 SOLO 两端使用
- **无需重复注册**：个人中心统一管理账号权限和任务记录
- **认证委托**：企业版通过 火山引擎 进行独立的认证和计费

### 3.3 设备身份管理

| 维度 | 说明 |
|------|------|
| **设备标识** | Android: AndroidID/OAID/GAID；iOS: IDFV |
| **客户端标识** | `vscode-{machineId}-{timestamp}` 格式（桌面端），移动端类似 |
| **Session ID** | 每次会话建立时生成，关联 WebSocket 连接 |
| **最多设备** | 同一账号最多 10 台设备同时在线，不互踢 |
| **设备配对** | 同账号自动发现，无需配对码 |

来源：[Trae 隐私政策](https://www.trae.com.cn/privacy-policy) §1.2 + [TraeUsage WebSocket协议](https://github.com/whyuds/TraeUsage/blob/master/WEBSOCKET_PROTOCOL.md)

## 四、数据同步架构

### 4.1 State Sync 核心理念

Trae Solo 的三端同步不是简单的"聊天记录漫游"，而是**State Sync（状态同步）**——同步"工作的逻辑原点"：

> "Agent 始终清晰地掌握着当前的目标、已达成的共识、下一步的执行边界，以及哪些关键节点需要人类的实时决策。"

来源：[ZPedia - State Sync 分析](http://m.toutiao.com/group/7636989712791552512/)

### 4.2 同步层级

```
Layer 3: 任务上下文同步
  ├─ 对话历史完整同步（不是摘要）
  ├─ 任务进度状态（pending/running/done）
  ├─ 文件产物（代码、PPT、文档等）
  └─ AI 决策记录

Layer 2: 代码仓库同步
  ├─ GitHub 集成 → 代码推送到 GitHub
  ├─ 跨设备通过 GitHub 拉取最新代码
  └─ 云端环境（Cloud Agent）独立副本

Layer 1: 文件系统同步
  ├─ 桌面端 Documents 目录 → 手机远程读写
  ├─ 飞书文档集成 → 文档存储飞书云端
  └─ 本地加密同步（信任设备间局域网）

Layer 0: 设备状态同步
  ├─ 在线/离线状态
  ├─ 配对设备列表
  └─ 防睡眠状态
```

### 4.3 数据流向

```
                     ┌──────────┐
                     │ Trae 云端 │
                     │  (北京引力  │
                     │   弹弓)    │
                     └────┬─────┘
          ┌───────────────┼───────────────┐
          ↓               ↓               ↓
    ┌──────────┐   ┌──────────┐   ┌──────────┐
    │ 移动端    │   │ 桌面端    │   │ Web 端   │
    │ State:   │   │ State:   │   │ State:   │
    │ 发起任务  │←→│ 执行任务  │←→│ 查看进度  │
    └──────────┘   └──────────┘   └──────────┘
```

**流程说明**：
1. 手机下发任务 → 云端接收 → 广播给配对桌面端
2. 桌面端执行 → 实时推送进度到云端 → 手机/Web 实时更新
3. 产物（代码/文档）→ GitHub 或飞书云端存储 → 各端可访问
4. 设备离线 → 任务降级到云端执行，保证不中断

### 4.4 GitHub 集成同步

根据隐私政策：

> "因本服务通过连接 GitHub 来实现跨设备任务同步，你的代码库将自动推送到你的 GitHub 账号进行管理"

来源：[Trae 隐私政策](https://www.trae.com.cn/privacy-policy)

## 六、核心功能对应的架构实现

### 6.1 设备配对（Phone Pairing）

```
桌面端设置 → 开启 "SOLO Mobile" 开关
                ↓
         桌面端向云端注册：我支持远程控制
                ↓
手机端 → 头像 → 设备管理 → 添加设备
                ↓
         云端返回同账号在线设备列表
                ↓
         手机选择目标电脑 → 配对成功
                ↓
         设备间建立 WebSocket 通道（云端中继）
```

**安全约束**：
- 只能访问 Documents 目录下的文件
- 需手动授权工作目录
- 支持文件夹级访问控制

### 6.2 远程操控电脑

```
手机输入指令
    ↓
云端接收 → 转发给配对桌面端
    ↓
桌面端 SOLO Desktop 接收指令
    ↓
启动 Agent 执行（读写文件/运行脚本/终端操作）
    ↓
执行日志流式回传 → 云端 → 手机实时展示
    ↓
结果产物 → GitHub/飞书 → 各端同步
```

### 6.3 多任务并行

```
┌──────────────┐     ┌──────────────────────────────┐
│  任务队列     │     │  云端/桌面执行引擎              │
│              │     │                              │
│  Task A ─────┼────→│  桌面端 1 执行                │
│  Task B ─────┼────→│  云端执行                    │
│  Task C ─────┼────→│  桌面端 2 执行                │
│              │     │                              │
│  统一监控     │←────│  实时状态汇总                   │

## 七、与你需求的对照分析

### 7.1 可复用架构点

| Trae Solo 架构 | 你的需求 | 复用程度 |
|---------------|---------|---------|
| 双智能体架构 (Builder + Coder) | 手机端 Builder 拆解任务，桌面端 Claude Code 执行 | ✅ 直接复用思路 |
| State Sync 多端同步 | 手机 ↔ 桌面端对话同步 | ✅ 可复用同步模型 |
| 设备配对 + 免配对码 | 手机连桌面 | ✅ 直接复用 |
| WebSocket 实时通信 | 手机 ↔ clwd-on-desk Bridge | ✅ 技术栈一致 |
| 手机纯对话 UI | 你只需要对话式交互 | ✅ 设计参考 |
| JWT + OAuth 认证 | 身份识别 | ✅ 直接复用 |
| GitHub 数据同步 | 代码/任务同步 | ✅ 可套用 |

### 7.2 不适用点

| Trae Solo 特性 | 不适用的原因 |
|---------------|------------|
| 字节云端 Agent | 你需要 Claude Code，不是豆包模型 |
| 飞书集成 | 你可能不需要 |
| SOLO Coder | 替换为 Claude Code CLI |
| 火山引擎认证 | 替换为你自己的认证 |

### 7.3 你自己的实现可以借鉴的架构

```
手机: PWA/React Native Chat UI
  ├─ 输入: 文本 + 语音
  ├─ 展示: 对话气泡 + 审批卡片
  └─ 通信: WebSocket → Bridge Server

Bridge Server（桌面端）
  ├─ clwd-on-desk 扩展
  ├─ 接收手机消息 → 转给 Claude Code
  ├─ 拦截 Claude Code 审批 → 推手机
  └─ 流转执行日志 → 手机展示

Claude Code CLI（桌面端）
  ├─ 标准输入/输出驱动
  ├─ 真正执行代码/命令/文件操作
  └─ 审批通过 stdout 回调
```

## 八、关键安全与隐私设计

| 维度 | Trae Solo 方案 |
|------|---------------|
| **数据存储** | 本地文件不存储云端（Web 端除外）；代码通过 GitHub 同步 |
| **文件隔离** | 桌面端仅授权目录可被 Agent 访问 |
| **传输加密** | HTTPS + WSS 加密 |
| **设备管理** | 用户可在设备管理页面移除/解绑设备 |
| **隐私基础** | 收集设备标识符用于安全风控，不存储实际代码库文件到字节服务器（桌面端） |

来源：[Trae 隐私政策](https://www.trae.com.cn/privacy-policy)

## 十、结论

Trae Solo 移动端本质上是一个**将 AI Agent 能力延伸到手机的远程控
制终端**，而非移动端 IDE。它的架构核心价值在于：

1. **身份统一**：一个账号打通三端，自动发现设备
2. **State Sync**：不只是数据同步，是 Agent 执行上下文的实时同步
3. **双智能体分离**：规划在云端/手机，执行在桌面端
4. **跨端接力**：任务在手机上发起，桌面端执行，Web 端查看

这些设计可以直接指导你的"手机纯对话 + Claude Code 桌面端执行"方案的架构设计。
