## 二、逐项评审

### 2.1 完整性（22/25）

#### ✅ user-goals-covered — 通过

- 四类用户画像（离开桌面的开发者/创业者/多任务并行者/灵感驱动者）精准匹配原始意图
- 5 个核心场景（S1-S5）覆盖了通勤下发、审批、进度查看、增量优化、多项目监控
- §1.3 核心价值主张直接回应用户原始需求中的每一个痛点

#### ✅ no-assumption-only-path — 通过

- 交互模式不走"手机 IDE"假设，确实按用户要求只做纯对话
- 明确声明不包含代码编辑器（§11）—— 这恰好是用户明确提出的要求
- 双端通信设计考虑了局域网和互联网两种路径

#### ✅ success-criteria-defined — 通过

- §12 定义了 6 项可量化指标，均含测量方式
- 指标贴近 MVP 实际（消息延迟 < 300ms、审批闭环 < 2s）

#### ✅ out-of-scope-defined — 通过

- §11 明确列出 9 项不做的内容，每条附有理由
- 非目标与用户原始需求一致（不造 IDE、不自研 AI 模型）

#### ⚠️ 建议

| ID | 问题 | 严重度 |
|----|------|--------|
| C-01 | **F-01.1 手机号注册需短信验证码，但 Bridge Server 由用户自部署，短信网关是一个重大外部依赖**。MVP 阶段建议降级为仅支持 GitHub OAuth | 中 |
| C-02 | **过渡动画、错误状态 UI、空状态页面未定义**。对话列表为空时、WebSocket 断连时、审批超时时的 UI 状态未描述 | 低 |
| C-03 | **缺少用户反馈路径**。用户如何报告问题？是否需要 Crashlytics/埋点？ | 低 |

#### ✅ feature-to-intent — 通过

- 所有 46 条功能需求（F-01 到 F-12）均可追溯到 `00-user-intent.md`：
  - "Claude Code 集成" → §4.3 全部交互功能 + §6.3.2 审批流转

- 例如：F-06.3 "一键允许/拒绝" 如何验收？多少秒内？触摸目标尺寸？

#### ✅ dependency-clarity — 通过

- §13.2 明确列出 5 个外部依赖（Claude Code CLI / clwd-on-desk / Node.js / Expo / Firebase）
- Phase 1/2/3 的时间线隐含了依赖顺序

#### ⚠️ 建议

| ID | 问题 | 严重度 |
|----|------|--------|
| T-01 | **每个功能模块缺少独立的验收条件**。建议为 F-01 到 F-12 各新增 "验收条件" 列 | 中 |
| T-02 | **F-10.3 "手机消息 → Bridge → Claude stdin" 的数据流缺少降级路径**：当 Claude Code 子进程崩溃时消息是否丢失？需定义行为 | 中 |

### 2.3 可行性（18/25）

#### ✅ tech-feasible — 通过

- React Native 0.76+ + Expo 是经过验证的移动端方案
- WebSocket + JWT 架构成熟，clwd-on-desk 已有 HTTP Server 基础
- Claude Code stdin/stdout 接管在终端模式已验证可行

#### ⚠️ unresolved-risks — 带条件通过

| 风险 | 当前缓解 | 不足 |
|------|---------|------|
| Claude Code 协议变更 | "关注 Release Notes" | 过于被动 |
| WebSocket 移动网络不稳定 | 重连机制 | 未定义消息去重/乱序处理 |

#### ⚠️ external-deps-confirmed — 带条件通过

- clwd-on-desk 的 Bridge Server 扩展是**本项目自建依赖**（非第三方），但它是 PRD 外部的前置条件
- Firebase 标注为 Phase 2 可选，但 §8.2 推送通知（F-12）是 P0 需求。矛盾：**P0 推送通知如何在不依赖 Firebase 的 MVP 阶段实现？**

#### ⚠️ 建议

| ID | 问题 | 严重度 |
|----|------|--------|
| F-01 | **F-12.1 "审批推送通知（App 在后台）" 标记为 P0，但推送方案标记为 Phase 3**。MVP 阶段 App 在后台时审批如何到达用户？建议：MVP 阶段通过 WebSocket 长连接实现前台推送，后台推送降级为 P2 | **高** |
| F-02 | **Bridge Server 的认证服务（短信/OAuth）需自建，成本未评估**。MVP 最小方案：仅支持 GitHub OAuth + 局域网配对码（无需云端账号体系） | 中 |
| F-03 | **"云端注册"（F-02.1）与 MVP 局域网定位矛盾**。局域网模式下设备发现需 mDNS/Bonjour 而非云端注册，PRD 未区分 | 中 |
