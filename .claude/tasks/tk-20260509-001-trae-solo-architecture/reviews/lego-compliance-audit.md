### 1.2 "子 Agent 可直接制造"硬规则

> 规范 §4.5 硬规则：方案详设的粒度必须达到"子 Agent 拿到设计稿后，不需要额外决策就能写出代码"。

按模块逐一判定：

| 模块 | 子 Agent 可制造？ | 缺口 |
|------|-----------------|------|
| CLIDriver (§7.2) | ✅ 有 spawn 参数 / stdio 处理 / exit 逻辑伪代码 | 无 |
| ApprovalEngine (§7.3) | ✅ 有完整伪代码（Map + Promise + 60s 超时） | 无 |
| WebSocket 客户端 (P7) | ✅ §5.5 完整类型定义 + §3.1 状态机 | 无 |
| ChatScreen (P9) | ✅ §2 组件树 + §4.2 ChatStore 接口 | 无 |
| SQLite 持久化 (P12) | ✅ §6.1 5 张表 DDL | 无 |
| AuthService (P4) | ⚠️ 函数签名有但缺 JWT 签发/验证细节 | 缺 key 管理 / HS256 参数 |
| REST API (P5) | ⚠️ 有端点 + HTTP method + 路径，但缺 Request/Response schema | 部分端点无 Body schema |
| Expo 脚手架 (P6) | ✅ 技术栈已选型 | 无 |
| AuthScreen (P8) | ⚠️ 有组件名但缺 OAuth 重定向 schema | 缺 deep link URI 格式 |

**结论：9/14 packets 可直接制造，5 个缺 L4 级细节。可通过硬规则但建议补全。**

### 1.3 §4.6.2 Agent 分配矩阵（10 字段）

规范要求每个 work packet 必须含 10 个字段，当前 §13.1 只有 5 列：

| 字段 | 规范要求 | 当前 §13.1 | 状态 |
|------|---------|-----------|------|
| `packet_id` | 是 | ✅ P1-P14 | 合规 |
| `lego_level` | L1/L2/L3/L4 | ❌ **缺失** | **违规** |
| `objective` | 单一明确目标 | ⚠️ 当前是模块名 | **不充分** |
| `timeout_minutes` | N (含默认范围) | ❌ **缺失** | **违规** |
| `read_scope` | 可读设计文档范围 | ❌ **缺失** | **违规** |
| `write_scope` | 可写代码范围 | ❌ **缺失** | **违规** |
| `dependencies` | 依赖的 packet_id | ✅ 有 | 合规 |
| `acceptance` | 怎么算完成 | ❌ **缺失** | **违规** |
| `verification` | 怎么验证 | ❌ **缺失** | **违规** |
| `rollback` | 如何回滚 | ❌ **缺失** | **违规** |

**结论：10 字段中缺 7 个。这是最重要的违规项。**

### 1.4 §4.6.5 L4 派发判定标准

> 规范要求：主线程在 Fan-Out 前必须对每个 L4 packet 应用判定。

| 规则 | Design 中是否有 | 状态 |
|------|---------------|------|
| < 50 行或 ≤ 3 次工具调用 → 主线程直接 | ❌ 无 | **缺失** |
| ≥ 50 行或需独立调研 → 子 Agent | ❌ 无 | **缺失** |
| 安全敏感代码 → 必须子 Agent | ❌ 无 | **缺失** |
| 强依赖 L4 → 优先主线程 | ❌ 无 | **缺失** |
| 批量同类 ≥ 3 → 子 Agent | ❌ 无 | **缺失** |

**结论：完全缺失。**
