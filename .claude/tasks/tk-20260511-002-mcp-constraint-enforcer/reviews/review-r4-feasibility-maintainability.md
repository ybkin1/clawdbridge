# Review Report: Feasibility + Maintainability

## Verdict: Fail

> **Rationale**: The MCP constraint-enforcer is operationally feasible for trivial/single-developer workloads but structurally unsuited for sustained maintenance by a rotating team. The config cache hot-reload works in principle but is undermined by per-call inefficiency. There is zero token-estimation logic despite being a stated dimension. Config drift is guaranteed because `config-sync-check.js` only validates 5 hardcoded dimensions (not "all 5 config files" as claimed), has no CI integration, and cannot detect schema-level mismatches between YAML configs and the `evaluateCondition` engine. Adding a new phase requires coordinated edits across at least 6 files with no automation. Most critically, there is no schema definition for any config file — the "machine-readable projection" is actually human-maintained free-form YAML with implicit contracts enforced only by runtime `switch` cases.

## Findings

### Feasibility

| Severity | ID | Description | Evidence | Remediation |
|----------|----|-------------|----------|-------------|
| Critical | F-001 | **Config cache hot-reload logic is correct but operationally weak.** `loadConfig` compares `mtimeMs` per file (`enforcer.js:50-52`), but it is invoked **4+ times per tool call** with no directory-level batching. If two configs change within the same millisecond (common on `git checkout` or bulk edit), the cache may serve stale data for one file while reloading another, producing an inconsistent config snapshot. | `enforcer.js:46-61` | Batch all configs into a single `loadAllConfigs()` call keyed by the directory's max mtime; or use a single global generation counter incremented on any mtime change. |
| High | F-002 | **`config-sync-check.js` validates only 5 hardcoded checks, not "all 5 config dimensions" as its header claims.** The checks are: (1) mechanical-conditions core IDs, (2) phase-state-machine gates mapping, (3) registry checker_refs against index, (4) write-permissions sensitive patterns, (5) mcp-capabilities manual_gate_pending_policy. It does **NOT** validate: `mcp-capabilities.automatable` list against `evaluateCondition` switch cases, `write-permissions.roles` against any consumer, `phase-state-machine.next` linkage consistency (e.g. no dangling next), or `registry.yaml` version field against anything. | `config-sync-check.js:44-167` | Expand the checker to perform cross-dimensional validation: automatable IDs vs. `evaluateCondition` cases, role patterns vs. `validateWritePermission` consumers, next-linkage acyclicity, and registry version consistency. |
| Critical | F-003 | **`config-sync-check.js` has a fragile dependency on `js-yaml` installed inside the MCP server's `node_modules`.** It uses `createRequire` to reach `../mcp-servers/constraint-enforcer/node_modules/js-yaml/index.js`. If the MCP server dependencies are not installed (e.g. fresh clone, `npm ci` not run), the checker crashes with `MODULE_NOT_FOUND`. There is no fallback to a globally installed `js-yaml` or graceful degradation. | `config-sync-check.js:10-14` | Add a fallback resolution chain: try global `js-yaml`, try `npm root -g`, then fail with a human-readable error. Or vendor a minimal YAML parser into `.claude/checkers/`. |
| Critical | F-004 | **There is no token estimation logic anywhere in the codebase.** The feasibility audit dimension asks "Is there any token estimation logic at all?" The answer is definitively **no**. Neither `enforcer.js`, `config-sync-check.js`, nor any config file contains references to token counting, context budget calculation, or API cost estimation. The `context_budget >= 55%` scope rule in `registry.yaml` is evaluated numerically but the actual budget is never computed by the MCP. | Full codebase grep for `token`, `budget`, `estimate`, `cost` in `.claude/mcp-servers/constraint-enforcer/` and `.claude/checkers/` | Implement a `token-estimation` utility module (even if approximate: char count / 4) and expose it via MCP tool or config field. Without it, the `context_budget` scope rule is a dead letter. |
| High | F-005 | **A single developer cannot realistically maintain 4 YAML configs + contracts without drift because the configs are implicitly coupled to `evaluateCondition` switch cases.** When `mechanical-conditions.yaml` adds a new `check_type`, the developer must remember to add a `case` to `enforcer.js:117-256`. There is no static analysis, no code generation, and no test that fails when they forget. | `enforcer.js:117-256` switch statement; `mechanical-conditions.yaml` free-form `check_type` strings | Generate the `evaluateCondition` switch from `mcp-capabilities.yaml` automatable list at build time; or add a runtime assertion that every `check_type` in loaded configs has a matching handler, and fail loudly if not. |
| High | F-006 | **Config file schemas are not documented anywhere in machine-readable form.** Each `.yaml` has human comments describing fields (e.g. `mechanical-conditions.yaml:8-17`), but there is no JSON Schema, Zod, Joi, or TypeScript interface. The "schema" is the `evaluateCondition` switch cases and scattered `?.` accesses in `enforcer.js`. A typo like `blocker_leval` would silently produce `undefined` and be treated as non-blocking. | Absence of any schema file in `.claude/config/` or `.claude/contracts/` | Add a `config-schema.yaml` or JSON Schema for each config dimension; validate configs against it at `loadConfig` time. |

### Maintainability

| Severity | ID | Description | Evidence | Remediation |
|----------|----|-------------|----------|-------------|
| Critical | M-001 | **`config-sync-check` has no CI integration path.** The `.github/workflows/` directory only contains `build-apk.yml` (Android build). There is no GitHub Action, pre-commit hook, or npm script that runs `config-sync-check.sh`. It is purely manual. A developer can edit configs, commit, push, and break the MCP without any automated signal. | `.github/workflows/build-apk.yml`; absence of any workflow referencing `config-sync-check` | Add a `.github/workflows/config-consistency.yml` that runs `bash .claude/checkers/config-sync-check.sh` on every PR touching `.claude/config/`, `.claude/contracts/registry.yaml`, or `.claude/checkers/index.yaml`. |
| Critical | M-002 | **When a new `check_type` is added to `mechanical-conditions.yaml` but `evaluateCondition` doesn't support it, the system silently passes.** The `default` case at `enforcer.js:251-255` returns `{ passed: true, gap: null }`. This is a catastrophic silent failure: a newly added security or compliance condition is simply ignored. | `enforcer.js:251-255` | Change the `default` case to return `{ passed: false, gap: \`Unknown check_type: ${checkType}\` }`. Additionally, add a startup validation that loads all configs and asserts every `check_type` is in the known set. |
| High | M-003 | **Config files have a `version` field, but it is never read or enforced.** All 4 configs declare `version: 1`, yet `loadConfig`, `evaluateCondition`, and `config-sync-check.js` never inspect it. If `mechanical-conditions.yaml` is bumped to `version: 2` with a breaking schema change, the MCP will parse it blindly and likely crash or misbehave. | `mechanical-conditions.yaml:19`, `phase-state-machine.yaml:7`, `write-permissions.yaml:7`, `mcp-capabilities.yaml:7`; no version check in `enforcer.js` or `config-sync-check.js` | Implement a `min_supported_version` and `max_supported_version` in the MCP; reject configs whose version is outside the supported range with a clear error. |
| High | M-004 | **Contract docs (`.md`) and config files (`.yaml`) are not kept in sync automatically.** The YAML headers claim they are "来源：task-tracking-workflow-spec.md §X", but there is no tooling to verify this. For example, `task-tracking-workflow-spec.md §4.3` defines 10 phases with specific gates, and `phase-state-machine.yaml` mirrors them — but `config-sync-check.js` only validates the gate mapping, not whether the phase list itself is complete or whether `next` pointers form a DAG. | `config-sync-check.js:75-108` | Add a structural validation pass: verify all `next` pointers resolve to existing phases, detect cycles, and ensure every phase in the spec has a YAML entry. |
| High | M-005 | **The cost of adding a new phase is high and error-prone.** A developer must update: (1) `phase-state-machine.yaml` (add phase + gates + next), (2) `task-tracking-workflow-spec.md §4.1` (state machine diagram), (3) `task-tracking-workflow-spec.md §4.3` (gate mapping table), (4) `config-sync-check.js` expected gates object (`enforcer.js` is not the only place — the JS checker hardcodes the same mapping at line 80-91), (5) `mechanical-conditions.yaml` if the phase has special exemptions (e.g. `clarify` exempts `evidence_lock_exists`), (6) any contract referencing phase lists (e.g. `registry.yaml` effective_scope). That is **at least 6 files**, two of which are in different languages (YAML + Markdown + JavaScript). | `phase-state-machine.yaml`; `task-tracking-workflow-spec.md:150-163`; `config-sync-check.js:80-91`; `mechanical-conditions.yaml:96` | Generate `config-sync-check.js` expected gates and the spec table from `phase-state-machine.yaml` via a build script; keep phase metadata in one YAML file and derive all downstream artifacts. |
| Medium | M-006 | **`evaluateEffectiveScope` silently skips unknown syntax rather than failing.** At `enforcer.js:1018-1021`, bare domain strings that don't match any known pattern are treated as "must match literally against context", and if the context key is undefined, it returns `false`. However, a typo in scope syntax (e.g. `phase=bulid` instead of `build`) would simply cause the contract to never match, with no warning. | `enforcer.js:978-1025` | Add a strict mode for `evaluateEffectiveScope` that logs warnings for unrecognized scope expressions; or maintain a whitelist of valid expression patterns and reject unknowns. |
| Medium | M-007 | **`config-sync-check.js` swallows YAML parse errors.** Both `loadYaml` and `extractYamlFromMarkdown` return `null` on any exception without logging the error message. A malformed config file is indistinguishable from a missing file, making debugging difficult. | `config-sync-check.js:18-25`, `27-40` | Log the specific parse error (at least to stderr) before returning `null`; include the exception message in the error array. |
| Medium | M-008 | **The `registry.yaml` embedded-in-markdown format is a maintenance hazard.** `loadRegistryYaml` and `extractYamlFromMarkdown` rely on a regex ``/```yaml\s*\n([\s\S]*?)\n```/`` to extract YAML from Markdown. If a contract author adds a second YAML block, or changes the fence to ` ```yaml ` with trailing spaces, the regex may match the wrong block or fail. | `enforcer.js:904-923`; `config-sync-check.js:27-40` | Store `registry.yaml` as a pure YAML file, not embedded in Markdown. If Markdown wrapping is required for human readability, use a front-matter format (YAML between `---` delimiters at the top) which is unambiguous. |

## Summary

The MCP constraint-enforcer suffers from a fundamental tension: it aspires to be a **configuration-driven, zero-hard-code engine** (a laudable goal), but the configuration layer lacks the structural rigor to sustain that claim. The "machine-readable projection" is actually a collection of free-form YAML files with implicit schemas enforced only by runtime `switch` statements and manual cross-checks in `config-sync-check.js`.

**Critical systemic risks:**
1. **Silent default pass** (`evaluateCondition` `default` case) makes the enforcer unsafe by design. Any new condition type is automatically ignored rather than blocking.
2. **No token estimation** renders the `context_budget` scope rule inoperable, undermining a core governance mechanism.
3. **No CI for config-sync-check** means config drift will reach production (or at least active Agent sessions) before it is detected.
4. **High phase-addition cost** (6 files, 3 languages) guarantees that the phase state machine will stagnate or become inconsistent as the project evolves.

**Immediate actions required:**
1. **Fail loud on unknown `check_type`** — change `default` to `passed: false`.
2. **Add startup config validation** — assert every `check_type` in loaded configs has a handler.
3. **Add CI workflow** for `config-sync-check.sh` on PRs touching `.claude/config/` or `.claude/contracts/registry.yaml`.
4. **Generate `config-sync-check.js` expectations** from `phase-state-machine.yaml` instead of hardcoding them.
5. **Implement minimal token estimation** (chars/4) so `context_budget` rules can function.
6. **Add JSON Schema or Zod validation** for all config files, enforced at `loadConfig` time.
7. **Read and enforce `version` fields** in configs; reject unsupported versions.

Without these changes, the constraint-enforcer will become a liability: it gives the illusion of rigorous mechanical governance while silently ignoring conditions, permitting drift, and imposing an unsustainable maintenance burden on the规范架构维护者.
