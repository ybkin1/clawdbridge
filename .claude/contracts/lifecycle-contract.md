# 生命周期契约 (Lifecycle Contract)

> 版本: v1.0
> 状态: draft-ready
> 归属: L3 专项契约
> 依赖: runtime-execution-spec.md (L1)
> 设计原则: Phase 0-6 全生命周期定义，支持轻量/标准/完整三档文档深度

---

## 1. 设计目标

将 `Claude-Bridge-Project-Standards-v1.0.md` 中的生命周期管理规范提炼为独立契约，明确：
- 每个阶段的输入、输出、负责人、执行过程
- 各阶段检查单与通过标准
- 文档深度分层（L1/L2/L3）与公开性策略
- 角色模式定义（/pm /tech-lead /pmo /dev）

---

## 2. 角色模式定义

| 角色 | 激活指令 | 核心职责 | 决策禁区（必须由人类做出） |
|------|---------|---------|------------------------|
| **/pm** | `/pm` 或 `@product-manager` | 需求洞察 → PRD编写 → 验收标准制定 → 竞品分析 | 产品愿景、MVP范围裁剪、商业模式、目标用户最终确认 |
| **/tech-lead** | `/tech-lead` 或 `@tech-lead` | 架构守护 → 技术选型 → 代码规范 → 技术债务管理 | — |
| **/pmo** | `/pmo` 或 `@project-manager` | 计划制定 → 进度跟踪 → 风险管控 → 发布管理 | 变更审批(Go/No-Go)、发布审批、资源冲突仲裁 |
| **/dev** | `/dev` 或默认模式 | 编码实现 → 单元测试 → 代码审查 → 文档更新 | — |

**角色切换规则**：
- 禁止同一轮对话内频繁切换角色（防止上下文污染）
- 跨角色协作顺序：/pm → /tech-lead → /pmo → /dev

---

## 3. 文档深度分层

| 深度 | 页数 | 适用场景 | 公开性 |
|------|------|---------|--------|
| **L1** | 1-2 页 | 个人 MVP、技术验证、4 周内交付 | 内部 |
| **L2** | 10-30 页 | 团队协作、对外发布、1-3 个月 | 内部 |
| **L3** | 完整版 + 汇报材料 | 向领导汇报、商业化、3 个月+ | 对外 |

**模式选择规则**：
- 一经选定，整个项目周期内不得随意切换
- Claude Bridge 建议：Phase 1 采用轻量模式，Phase 3（公网发布）起切换至标准模式

## 4. Phase 0-6 全生命周期定义

### 4.1 Phase 0: 初期调研与学习分析

**目标**: 通过联网深度搜索和竞品分析，形成对问题域的深度理解。

**负责人**: /pm 主导，/tech-lead 辅助技术可行性判断

**输入**: 用户原始需求描述、已知竞品名称、目标用户画像草图

**执行过程**:
1. 激活 /pm，输入用户需求
2. 联网搜索竞品（至少 3 个）的技术实现、用户评价、已知缺陷
3. 联网搜索目标技术栈最新最佳实践
4. 深度分析：竞品功能矩阵、差异化机会、技术风险点
5. 输出《学习分析报告》

**输出物**:

| 文档 | 深度 | 公开性 | 模板 |
|------|------|--------|------|
| 学习分析报告 | L2（5-10 页） | 可对外 | `templates/01-research-report-template.md` |
| 竞品分析报告 | L2（3-5 页） | 内部 | `templates/01-research-report-template.md`（竞品章节） |

**检查单（Phase 0 出口）**:
- [ ] 是否调研了至少 3 个竞品？
- [ ] 是否覆盖了目标技术栈的最新最佳实践？
- [ ] 是否识别了至少 2 个已知技术陷阱？
- [ ] 是否明确了对本项目的启示和待验证项？
- [ ] 所有引用是否有可追溯的信息来源？

**通过标准**: 检查单全部勾选，且发现至少 1 个此前未知的重大风险或机会。

---

### 4.2 Phase 1: 产品定义（PRD）

**目标**: 将用户需求转化为结构化的产品需求文档，作为后续所有设计和开发的唯一真理来源。

**负责人**: /pm

**输入**: Phase 0 学习分析报告、竞品分析报告、用户原始需求

**执行过程**:
1. 激活 /pm
2. 基于调研结果，定义用户故事（US-1 ~ US-N）
3. 定义功能需求优先级（P0/P1/P2）
4. 定义非功能需求（NFR）：性能、兼容性、安全、可访问性
5. 定义目标设备矩阵（必须包含 iOS + 鸿蒙 + 至少 1 款 Android）
6. 输出 PRD（根据模式选择 L1/L2/L3）

**输出物**:

| 文档 | 深度 | 公开性 | 模板 |
|------|------|--------|------|
| PRD | L1/L2/L3 | 内部 | `templates/02-PRD-template.md` |
| 产品提案（如需向领导汇报） | L3 | 对外 | `templates/02-PRD-template.md`（Executive 章节） |

**检查单（Phase 1 出口）**:
- [ ] 每个用户故事是否有明确的验收标准（可测试）？
- [ ] P0 功能是否覆盖了核心价值主张？
- [ ] 目标设备矩阵是否包含 iOS + 鸿蒙？
- [ ] 是否定义了明确不做清单（防止范围蔓延）？
- [ ] 是否识别了关键假设和风险？
- [ ] 需求是否可追溯（每条需求有唯一 ID）？

**通过标准**: 检查单全部勾选，且 P0 功能 ≤ 5 个（MVP 原则）。

---

### 4.3 Phase 2: 方案详设

**目标**: 将 PRD 中的需求转化为可执行的技术方案，包括架构设计、接口定义、安全设计。

**负责人**: /tech-lead 主导，/pm 辅助确认需求理解无误

**输入**: PRD（L1/L2/L3）、技术决策记录（Phase 0 产出）

**规范依赖**: 对于 Standard/Complex 任务，必须加载并遵守 `standards/recursive-decomposition-spec.md`。方案详设阶段是递归拆解的主战场（L1→L2→L3→L4），必须遵循：
- 锚链机制：每个子组件设计文档头部必须携带溯源锚链
- 黄金约束提取：从 PRD 中提炼 3-5 条不可变约束
- 跨层评审：L3/L4 设计完成后必须执行 Cross-Level Review
- 兄弟一致性检查：同一父节点下的子组件必须进行设计一致性校验

**执行过程**:
1. 激活 /tech-lead，输入 PRD
2. 架构设计：模块划分、状态机定义、数据流设计
3. 接口设计：消息协议、API 契约、错误码定义
4. 安全设计：认证、授权、输入过滤、路径安全
5. 兼容性设计：iOS PWA 限制、鸿蒙 ArkWeb 特性、Windows node-pty
6. 输出《架构设计文档》+ 《接口协议文档》

**输出物**:

| 文档 | 深度 | 公开性 | 模板 |
|------|------|--------|------|
| 架构设计文档 | L2（10-20 页） | 内部 | `templates/03-arch-design-template.md` |
| 接口协议文档 | L2（精确定义） | 内部 | `templates/03-arch-design-template.md`（接口章节） |
| 安全设计文档 | L2（5-10 页） | 内部 | `templates/03-arch-design-template.md`（安全章节） |

**检查单（Phase 2 出口）**:
- [ ] 架构图是否覆盖了 PRD 中所有 P0 功能？
- [ ] 状态机是否有死循环或无法到达的状态？
- [ ] 接口协议是否有明确的版本号？
- [ ] 错误码是否完整（覆盖所有异常路径）？
- [ ] 安全设计是否覆盖了输入过滤、路径安全、认证授权？
- [ ] 兼容性设计是否明确包含 iOS + 鸿蒙的具体适配点？
- [ ] 是否识别了技术债务并制定了偿还计划？
- [ ] 所有设计决策是否有 ADR 记录？

**通过标准**: 检查单全部勾选，且架构设计通过了至少 1 轮同行评审。

---

### 4.4 Phase 3: 开发计划

**目标**: 将设计方案转化为可执行、可跟踪的开发计划。

**负责人**: /pmo

**输入**: 架构设计文档、PRD 中的里程碑定义

**执行过程**:
1. 激活 /pmo
2. WBS 分解：将架构模块拆分为可执行的任务（≤3 天/任务）
3. 依赖关系梳理：识别任务间的先后依赖
4. 里程碑定义：每阶段有可演示的产出
5. 风险识别：技术风险、进度风险、外部依赖
6. 输出《开发计划》+ 《风险登记册》

**输出物**:

| 文档 | 深度 | 公开性 | 模板 |
|------|------|--------|------|
| 开发计划 | L2（WBS + 时间线） | 内部 | `templates/04-dev-plan-template.md` |
| 风险登记册 | L2 | 内部 | `templates/04-dev-plan-template.md`（风险章节） |

**检查单（Phase 3 出口）**:
- [ ] 任务分解是否足够细（≤3 天/任务）？
- [ ] 依赖关系是否明确，无循环依赖？
- [ ] 每个里程碑是否有可验证的交付物和验收标准？
- [ ] 是否识别了 Top3 风险并有缓解措施？
- [ ] 计划是否预留了 20% 缓冲时间？

**通过标准**: 检查单全部勾选，且关键路径上的任务有明确的负责人和时间约束。

---

### 4.5 Phase 4: 编码实现

**目标**: 按照开发计划高质量完成代码开发。

**负责人**: /dev（默认模式）

**输入**: 架构设计文档、接口协议文档、开发计划

**执行过程**:
1. 激活 /dev
2. 按照开发计划逐项实现
3. 每完成一个模块，自测（单元测试 + 手动验证）
4. 提交代码（遵循 Git 工作流和 Commit 规范）
5. 触发 CI 流水线

**编码规范**（强制）：
- Simplicity First：最小代码解决问题
- Surgical Changes：只修改必要代码
- Goal-Driven：每个变更对应可验证的成功标准
- TypeScript：显式返回类型、禁止 `any`、优先 `interface`
- 错误处理：统一错误类层次、异步错误必须捕获
- 提交规范：Conventional Commits（`feat(scope): subject`）

**输出物**:

| 产出 | 说明 |
|------|------|
| 代码 | 符合编码规范的源代码 |
| 单元测试 | 覆盖率 ≥80%（核心模块） |
| Commit 历史 | 符合 Conventional Commits |

**检查单（Phase 4 出口，每个 PR）**:
- [ ] 代码是否通过了 ESLint + TypeScript 编译？
- [ ] 是否包含单元测试？核心模块覆盖率是否 ≥80%？
- [ ] 是否引入了新的依赖？如有，是否有 ADR 记录？
- [ ] 是否修改了接口协议？如有，协议文档是否同步更新？
- [ ] 是否处理了所有边界条件和错误路径？
- [ ] 是否进行了手动验证（Happy Path）？

**通过标准**: CI 全部通过 + Code Review 通过。

---

### 4.6 Phase 5: 测试验证

**目标**: 系统化验证产品质量，确保满足 PRD 中的验收标准。

**负责人**: /dev + /pm（联合）

**输入**: PRD 验收标准、测试方案、测试用例

**执行过程**:
1. 基于 PRD 验收标准，生成测试用例
2. 执行自动化测试（单元测试 + E2E 测试）
3. 执行手动测试（核心流程、边界条件、设备兼容性）
4. 记录缺陷，评估严重程度
5. 输出《测试报告》

**输出物**:

| 文档 | 深度 | 公开性 | 模板 |
|------|------|--------|------|
| 测试方案 | L2 | 内部 | `templates/05-test-plan-template.md` |
| 测试用例 | L2 | 内部 | `templates/05-test-plan-template.md`（用例章节） |
| 测试报告 | L2 | 内部 | `templates/05-test-plan-template.md`（报告章节） |

**检查单（Phase 5 出口）**:
- [ ] 测试用例是否覆盖了所有 P0 功能的验收标准？
- [ ] 单元测试覆盖率是否 ≥80%（核心模块）？
- [ ] E2E 测试是否覆盖了核心用户流程（US-1 ~ US-N）？
- [ ] 是否进行了真机测试（iOS + 鸿蒙）？
- [ ] 严重缺陷是否已全部修复？
- [ ] 测试报告是否给出了明确的 Go/No-Go 结论？

**通过标准**: 严重缺陷 = 0，且测试报告结论为 "Go"。

### 4.7 Phase 6: 发布与变更管理

**目标**: 安全、可控地将产品交付给用户，并管理后续变更。

**负责人**: /pmo 主导，/pm 辅助

**输入**: 测试报告（Go）、PRD 中的发布策略

**执行过程**:
1. 激活 /pmo
2. 准备发布材料：Changelog、版本号、回滚方案
3. 执行发布（遵循发布检查单）
4. 监控线上状态（错误率、性能指标）
5. 收集用户反馈

**输出物**:

| 文档 | 深度 | 公开性 | 模板 |
|------|------|--------|------|
| Changelog | L1 | 对外 | `templates/06-release-checklist.md`（Changelog 章节） |
| 发布检查单 | L1 | 内部 | `templates/06-release-checklist.md` |
| 变更请求单 | L1 | 内部 | `templates/06-release-checklist.md`（变更章节） |

**检查单（Phase 6 出口）**:
- [ ] 版本号已更新（SemVer）
- [ ] Changelog 已更新（Keep a Changelog）
- [ ] 文档已更新（用户手册、API 文档）
- [ ] 回滚方案已准备
- [ ] 线上监控已配置
- [ ] 通知渠道已准备（用户群、邮件）

**通过标准**: 检查单全部勾选，且线上监控无 Critical 告警。

**变更控制原则**:
- 任何对 PRD 的修改都必须通过变更请求单
- 变更请求必须经过影响分析
- 已发布的版本禁止直接修改，必须通过新版本发布

## 5. 评审体系

### 5.1 评审原则

1. **分层评审**: 不同深度对应不同评审严格度（轻量/标准/完整）
2. **角色分离**: 评审者不能是作者（AI 评审时，用不同 Agent 扮演评审者）
3. **检查单驱动**: 评审必须对照检查单逐项确认
4. **出口门禁**: 每个阶段的输出必须达到通过标准，才能进入下一阶段

### 5.2 各阶段评审检查单汇总

| 阶段 | 检查单 | 关键项数 | 执行者 | 模板路径 |
|------|--------|--------|--------|---------|
| Phase 0 | 调研评审检查单 | 4 | /pm | `checklists/product-review-checklist.md`（调研章节） |
| Phase 1 | PRD 评审检查单 | 6 | /pm + /tech-lead | `checklists/product-review-checklist.md` |
| Phase 2 | 设计评审检查单 | 8 | /tech-lead + /pm | `checklists/tech-review-checklist.md` |
| Phase 3 | 计划评审检查单 | 5 | /pmo | `checklists/plan-review-checklist.md` |
| Phase 4 | 代码评审检查单 | 6 | /tech-lead | `checklists/code-review-checklist.md` |
| Phase 5 | 测试评审检查单 | 6 | /pmo + /pm | `checklists/test-review-checklist.md` |
| Phase 6 | 发布评审检查单 | 6 | /pmo | `checklists/release-checklist.md` |

### 5.3 评审通过标准

| 等级 | 标准 | 后续动作 |
|------|------|---------|
| **通过** | 检查单全部勾选，无 Critical 问题 | 进入下一阶段 |
| **有条件通过** | 检查单全部勾选，有 Major 问题但已制定修复计划 | 进入下一阶段，Major 问题必须在下一里程碑前修复 |
| **不通过** | 有未解决的 Critical 问题，或检查单未全部勾选 | 打回修改，重新评审 |

## 6. 与现有机制的边界

| 规范 | 位置 | 关系 |
|------|------|------|
| `AGENTS.md` | 项目根目录 | 提供技术栈与编码标准，本契约引用其角色定义 |
| `runtime-execution-spec.md` | `.claude/project/standards/` | 运行时状态机，本契约声明各 Phase 的触发条件 |
| `review-gates-contract.md` | `.claude/project/contracts/` | 评审门禁三层 Gate，本契约声明何时调用 |

**边界声明**:
- 本契约不定义具体技术栈 → 由 `AGENTS.md` 持有
- 本契约不定义测试命令 → 由 `command-catalog.md` 持有
- 本契约不定义代码规范细节 → 由 `engineering-standards-contract.md` 持有
