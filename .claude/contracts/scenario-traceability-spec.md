# 场景追溯评审规范（Scenario Traceability Review Specification）

## 1. 目的

评审必须基于实际应用场景逐项走读代码，发现真实问题。禁止"看一眼说通过"式的流程合规评审。

## 2. 核心原则

评审的输入不是"代码有没有遵循设计文档"，而是"**在实际用户场景下，这段代码能不能跑通**"。

每个评审发现必须说明：
- **哪个场景** 出了问题
- **哪段代码** 有问题
- **什么性质** 的问题（代码错误 / 逻辑漏洞 / 功能缺失 / 接口不匹配 / 场景不适配）
- **影响什么功能** 或 **不适配什么场景**

## 3. 适用范围与场景界定

### 3.1 适用任务

本规范适用于所有 **Standard/Complex 任务**的编码评审（Stage 6）和测试评审（Stage 7）。Trivial 任务豁免。

### 3.2 不适用场景

- 纯文档评审（无代码变更）
- 纯配置变更（仅改配置文件，不涉及逻辑流）
- 纯重构（行为不变，仅内部结构调整）— 但需在追溯矩阵中标注"行为不变，仅验证回归"

### 3.3 分支与基线（Branch & Baseline）

评审开始前，评审者必须记录本次评审的 **diff 基线**：

| 字段 | 说明 | 示例 |
|------|------|------|
| `diff_base` | git diff 的对比基线（commit hash 或分支名） | `main` 或 `abc1234` |
| `review_branch` | 被评审的分支 | `feature/login` |
| `分支状态` | `fresh`（基于最新 main）/ `stale`（落后 main ≥ 5 commits） | `fresh` |

**规则**：
- 当 `分支状态=stale` 时，评审者必须在评审报告中标注"基于过时分支，结论可能被并发变更影响"
- 追溯矩阵中引用的代码位置必须以 `review_branch` 的实际内容为准，不以 `diff_base` 为准
- 若走读时发现目标文件已被其他开发者修改（与当前分支内容不一致），必须标注"并发冲突风险"

### 3.4 场景范围界定（Scope-Scoping）

场景追溯评审的代码阅读范围按**代码产出类型**分层，不同深度要求：

| 代码类型 | 定义 | 阅读深度 |
|---------|------|---------|
| **新产出代码** | 本次任务首次编写的函数/类/组件/API | **全量逐行走读**，无抽样 |
| **变更代码** | 已有代码的修改（逻辑变更、接口调整、条件修改） | **逐行读变更部分** + 受影响的上游调用方/下游被调用方（仅直接调用关系，不展开其调用链） |
| **未变更代码** | 之前已评审通过、本次未修改的存量代码 | 仅走读场景路径**必须经过**的部分，按调用链追溯到入口点，不逐行读无关逻辑 |

**范围划定步骤**：
1. 区分本次任务是"新产出"还是"迭代修改"
2. **新产出**：列出本次新增的全部代码文件，逐行走读每一行
3. **迭代修改**：
   - 从追溯矩阵的"场景路径"出发，列出涉及的代码文件
   - 标注哪些行是本次变更（通过 git diff 或作者提供的变更清单）
   - 对变更行逐行读
   - 对变更行的直接调用方/被调用方读关联逻辑（1 层 = 直接调用关系，不展开其调用链）
4. **未变更代码**：仅在场景路径必须经过时读取到入口点，不展开无关内部实现

**大规模变更（>20 文件）**：
- 按模块/组件分组追溯矩阵，不按文件平铺
- 每个模块选取代表性场景走读，不必覆盖所有文件的每一行
- 在评审报告中说明分组依据和抽样理由

**禁止行为**：
- 逐行阅读全量未变更代码（不是全库 code style review）
- 阅读与场景路径无关的工具函数/辅助类
- **对新增代码只抽样不逐行（抽样走读 = 浅层走读，触发 §7 硬失败）**
- 对变更代码只看 diff 不看上下文（必须结合调用方/被调用方理解变更影响）

## 4. 追溯矩阵

### 4.1 追溯链

```
需求来源 → 用户场景 → 场景路径 → 代码位置 → 测试用例
```

**需求来源**：可以是 PRD 文档、用户口头描述（整理为书面需求）、Issue/Ticket、commit message。评审者必须先将需求整理为可追溯的形式，再建立追溯链。

### 4.2 需求来源字段

追溯矩阵必须标注每个场景的需求来源：

| 字段 | 类型 | 约束 | 示例 |
|------|------|------|------|
| `需求来源` | enum | `PRD` / `口头描述` / `issue` / `commit` / `隐式` | `PRD` |
| `需求来源引用` | string | PRD 时为功能条目 ID；口头描述时为对话日期摘要；issue 时为 issue 编号 | `PRD-03` 或 `#GH-142` |

**无 PRD 时的处理**：当需求来源为 `口头描述` 或 `issue` 时，评审者必须先将需求整理为一句话需求描述，写入追溯矩阵的"场景名称"列，作为追溯链锚点。

### 4.3 层级定义

| 层级 | 定义 | 示例 |
|------|------|------|
| 需求 | 需求来源中的功能描述 | "用户可在手机上审批权限请求" |
| 场景 | 该需求下的具体用户操作路径 | 用户收到审批通知 → 点击打开 → 查看请求详情 → 点击批准 → 收到确认 |
| 场景路径 | 该场景的每一步操作 + 系统响应 | 步骤1: 手机收到 approval_request 消息 → 步骤2: 展示审批卡片 → ... |
| 代码位置 | 实现该路径步骤的具体代码文件和行号 | `AppViewModel.kt:142` 处理审批回调 |
| 测试用例 | 验证该路径的测试 | `AppViewModelTest.kt:testApprovalCallback()` |

### 4.4 追溯矩阵 Schema（严格约束）

评审者必须在 `tasks/<task-id>/reviews/scenario-traceability.md` 中填写追溯矩阵，字段必须符合以下约束：

**场景元数据表**：

| 字段 | 类型 | 约束 | 示例 |
|------|------|------|------|
| `场景ID` | `SC-<两位数字>` | 必须唯一，从 SC-01 起递增；跨任务引用时加任务前缀 `tk-<id>/SC-<两位数字>` | SC-01 |
| `需求来源` | enum（见 §4.2） | 必填 | `PRD` |
| `需求来源引用` | string（见 §4.2） | 必填 | `PRD-03` |
| `场景名称` | string | 简明描述，建议 ≤ 50 字，动宾结构 | "用户发送消息" |
| `场景类型` | enum | `happy_path` / `error_path` / `edge_case` / `security` / `operational` / `lifecycle` | `happy_path` |
| `优先级` | enum | `P0` / `P1` / `P2` | `P0` |
| `平台` | enum | `all` / `android` / `ios` / `pwa` / `harmonyos` / `server` / `desktop` / `cli` | `android` |

**追溯步骤表**：

| 字段 | 类型 | 约束 | 示例 |
|------|------|------|------|
| `步骤序号` | int | ≥ 1，递增（允许跳号以便插入步骤） | 1 |
| `用户操作` | string | 描述用户动作或系统触发事件 | "点击发送按钮" |
| `系统响应` | string | 描述系统预期响应 | "消息入队并发送" |
| `代码位置` | `<文件名>:<行号>` | 必须指向真实存在的代码文件；引用的行号必须在 git diff 涉及的变更文件范围内，或在场景路径直接调用链上 | `ChatScreen.kt:85` |
| `测试结果` | enum | `pass` / `fail` / `not_covered` / `not_applicable` | `pass` |
| `问题` | string 或 `—` | 无问题填 `—`；有问题引用问题编号（如 `#ISSUE-001`） | `#ISSUE-001` |

**问题记录表**：

| 字段 | 类型 | 约束 |
|------|------|------|
| `问题编号` | `ISSUE-<三位数字>` | 从 ISSUE-001 起递增 |
| `场景` | `SC-<两位数字>-步骤<序号>` | 必须引用场景ID和步骤序号 |
| `代码位置` | `<文件名>:<行号>` | 同上 |
| `问题类型` | enum（见 §5.2） | 必须为枚举值之一，不在枚举内时填 `其他` 并在问题描述中说明 |
| `严重程度` | enum | `Critical` / `Major` / `Minor` |
| `影响` | string | 描述影响的功能或不适配的场景 |
| `复现路径` | string | 用户操作步骤 |

### 4.5 追溯矩阵模板

```markdown
# 场景追溯矩阵 — tk-<id>

## 评审基线

- diff_base: main
- review_branch: feature/login
- 分支状态: fresh

## 场景元数据

| 场景ID | 需求来源 | 需求来源引用 | 场景名称 | 场景类型 | 优先级 | 平台 |
|--------|---------|-------------|---------|---------|--------|------|
| SC-01 | PRD | PRD-01 | 用户发送消息 | happy_path | P0 | android |
| SC-02 | PRD | PRD-01 | 断网重连后消息恢复 | error_path | P0 | android |
| SC-03 | 隐式 | — | 后台杀死后恢复 | lifecycle | P0 | android |

## 场景 SC-01: 用户发送消息

| 步骤 | 用户操作 | 系统响应 | 代码位置 | 测试结果 | 问题 |
|------|---------|---------|---------|---------|------|
| 1 | 输入文本并点击发送 | 文本校验通过 | ChatScreen.kt:85 | pass | — |
| 2 | — | 消息入队发送 | ChatViewModel.kt:142 | pass | — |

## 问题记录

### ISSUE-001 — 断网时消息状态不一致

- **场景**: SC-02-步骤3
- **代码位置**: MessageQueue.kt:67
- **问题类型**: 逻辑错误
- **严重程度**: Critical
- **影响**: 断网重连后消息重复发送
- **复现路径**: 断网 → 发消息（入队失败） → 重连 → 自动重试 → 重复发送
```

### 4.6 场景复用规则

同一功能在后续任务中被修改时，评审者可以复用之前任务的追溯矩阵：

| 情况 | 操作 |
|------|------|
| 本次变更不影响某场景的路径 | 复用原场景追溯结果，标注 `复用=tk-<id>/SC-<xx>`，仅验证变更行不影响该路径 |
| 本次变更修改了某场景的路径 | 复用原场景元数据，重新走读变更涉及的步骤 |
| 本次新增场景 | 按正常流程建立新场景追溯 |

**规则**：复用场景时，评审者必须验证原追溯矩阵引用的代码行号在当前分支仍然有效。无效引用 = 追溯矩阵不完整。

## 5. 评审走读规则

### 5.1 逐场景走读

评审者必须对每个场景：

1. **代入用户角色**：不是"这段代码写得好不好"，而是"用户这样操作，代码能不能跑通"
2. **按步骤走读**：从用户操作第一步开始，逐行读代码，跟踪数据流
3. **记录断点**：哪里跑不通、哪里数据丢失、哪里逻辑错误，记录下来
4. **标注影响**：该问题影响的是哪个场景、哪个功能

### 5.2 问题分类

| 问题类型 | 定义 | 默认严重度 | 示例 |
|---------|------|-----------|------|
| **场景不通** | 按场景路径走读，代码在某一步卡死/崩溃/返回错误 | Critical | NPE 导致崩溃、接口 500 |
| **逻辑错误** | 代码能跑通但行为错误（分支遗漏、状态不一致、竞态、数据不一致） | Critical | 条件判断遗漏 null 分支 |
| **功能缺失** | 场景要求的某一步没有代码实现 | Critical | PRD 要求"取消审批"但无代码 |
| **接口不匹配** | 上下游接口不一致（字段名/类型/可选性/协议版本） | Major | 前端传 `user_id` 后端读 `userId` |
| **场景不适配** | 代码在 A 场景能跑，B 场景不行（API 差异、网络抖动、后台杀死、生命周期差异） | Major | Android 12+ 通知权限未适配 |
| **边界遗漏** | 正常路径通，异常路径未处理（断网、超时、数据为空、并发） | Major | 列表为空时崩溃 |
| **体验降级** | 功能通了但体验差（加载无指示、错误无提示、操作无反馈） | Minor | 请求中无 loading 状态 |
| **资源泄漏** | 文件句柄/连接/内存未释放 | Major | 数据库连接未关闭 |
| **性能退化** | 功能正常但性能明显下降（全表扫描、N+1 查询、不必要的内存拷贝） | Major | 未使用索引的 WHERE 查询 |
| **其他** | 不在上述枚举中的问题类型 | 由评审者判定 | 无障碍不兼容、国际化缺失 | — 填 `其他` 时必须在问题描述中详细说明问题性质 |

### 5.3 场景目录要求（Minimum Scenario Catalog）

每个任务的追溯矩阵**至少**包含以下场景类别。是否需要某类别取决于本次任务变更涉及的产品组件：

| 变更涉及 | 最低场景要求 |
|---------|------------|
| 所有任务 | ≥1 个 P0 happy path（主流程） |
| 所有任务 | ≥1 个网络异常场景（断网/超时/弱网） |
| 所有任务 | ≥1 个异常输入场景（空数据/非法格式/越权） |
| 移动客户端（Android/iOS/HarmonyOS/PWA） | ≥1 个应用生命周期场景（冷启动/热启动/后台杀死恢复） |
| 移动客户端 | ≥1 个权限相关场景（拒绝/授予/运行时变更） |
| 移动客户端 | ≥1 个离线/在线切换场景 |
| 服务端（relay-server/FastAPI/MCP） | ≥1 个并发场景（多客户端同时连接） |
| 服务端 | ≥1 个资源耗尽场景（连接池满/内存/磁盘） |
| 涉及认证/授权/敏感数据的变更 | ≥1 个认证绕过尝试 |
| 涉及认证/授权/敏感数据的变更 | ≥1 个越权访问尝试 |
| 版本升级/数据迁移相关变更 | ≥1 个版本兼容场景（旧版客户端连接新版服务端 / 数据升级兼容） |

**裁剪规则**：如果本次变更只涉及单一组件（如仅修改 MCP server 代码），则只需满足该组件对应的场景要求，其他组件的场景要求不强制。但如果变更涉及跨组件交互，必须满足所有涉及组件的场景要求。

**未达到最低目录要求时，Professional Gate 不得 Pass。**

### 5.4 平台场景矩阵（Platform Scenario Matrix）

不同平台的同一场景可能有不同的生命周期和行为差异，必须在追溯矩阵中标注。

**移动客户端矩阵**：

| 场景 | Android | PWA (Web) | HarmonyOS |
|------|---------|-----------|-----------|
| 后台杀死恢复 | Activity 重建 + savedInstanceState | Service Worker 持久化 | Ability 状态恢复 |
| 通知权限 | Android 13+ 需运行时申请 | 浏览器通知 API | 鸿蒙通知权限 |
| 网络状态变化 | ConnectivityManager | navigator.onLine | Network Kit |
| 推送消息 | FCM / 厂商推送 | WebSocket 长连接 | 鸿蒙推送服务 |
| 文件存储 | Context.filesDir | IndexedDB / Cache API | 鸿蒙文件系统 |
| 冷启动 | Application.onCreate | Service Worker 激活 | Ability.onCreate |

**Harness 扩展矩阵**（Tauri 桌面客户端 + 服务端）：

| 场景 | Tauri Desktop | FastAPI 服务端 |
|------|--------------|---------------|
| 应用初始化 | Tauri app setup → Rust backend → WebView 渲染 | uvicorn worker 启动 → 连接池初始化 |
| 进程间通信 | Tauri commands (Rust ↔ JS) | REST/GraphQL endpoints |
| MCP 工具调用 | 通过 HTTP 调用 MCP server | MCP server 注册/发现/工具执行 |
| 事件通信 | NATS pub/sub 订阅 | NATS pub/sub 发布/响应 |
| 配置热更新 | Tauri config reload | 环境变量/配置中心 reload |
| 降级策略 | WebView fallback | 熔断/限流/降级中间件 |

**规则**：追溯矩阵中标注 `平台=all` 的场景，必须验证在上述关键差异点上是否有适配代码。若代码仅适配单一平台，必须在问题记录中标注"未适配平台 X 的场景差异"。

### 5.5 安全场景类

涉及以下代码变更时，必须在追溯矩阵中增加安全场景：
- 认证/授权逻辑变更
- 加密/解密逻辑变更
- 用户数据读写（PII）
- 网络通信协议变更（非 HTTPS 标准库封装）
- 文件上传/下载（涉及用户可控文件）
- 数据库操作涉及用户输入拼接或权限相关查询

安全场景必须验证：
- 输入是否经过校验（类型、长度、格式、SQL 注入、XSS）
- 敏感数据是否加密传输和存储
- 日志是否泄露敏感信息
- 权限校验是否完整（不仅前端，服务端必须二次校验）

### 5.6 运维场景类

所有服务端产品和移动客户端都必须考虑运维场景：
- **服务端**：服务启动/停止/重启、配置热更新、日志轮转、健康检查、降级策略、数据库连接池耗尽恢复
- **移动客户端**：版本升级兼容、首次启动引导、数据迁移、错误上报、离线数据持久化

### 5.7 发现问题时的报告格式

```markdown
### ISSUE-<编号> — 问题简述

- **场景**: SC-<ID> 步骤<N>: [场景描述]
- **代码位置**: <文件名>:<行号>
- **问题类型**: [场景不通/逻辑错误/功能缺失/接口不匹配/场景不适配/边界遗漏/体验降级/资源泄漏/性能退化/其他]
- **问题描述**: [具体哪里有问题，为什么错]
- **影响**: [影响什么功能/不适配什么场景]
- **复现路径**: [用户怎么操作能触发]
- **严重程度**: Critical / Major / Minor
```

## 6. 追溯矩阵真实性校验

追溯矩阵中的代码引用必须可验证：

| 校验项 | 判定标准 |
|--------|---------|
| 行号存在性 | 引用的 `文件:行号` 必须在 git diff 涉及的变更文件范围内，或在场景路径的直接调用链上（可通过 code-review-graph MCP 验证） |
| 变更对应性 | P0 场景的追溯步骤必须至少有 1 条代码位置对应本次变更（不能全部指向未变更代码） |
| 代码片段证据 | P0 场景的至少 2 个关键步骤应附带代码片段引用（≥3 行代码），而非仅写 `文件:行号` |

**任一校验项失败 → Professional Gate = Failed。**

## 7. 浅层走读硬失败条件

以下情况视为**浅层走读**，Professional Gate 直接 Fail：

| 浅层行为 | 判定标准 |
|---------|---------|
| 未读代码 | 追溯矩阵中"代码位置"列为空或只写了文件名无行号 |
| 新代码未逐行 | 新产出代码未全量逐行走读（只抽样或部分阅读） |
| 变更无上下文 | 对变更代码只看 diff 行，未读调用方/被调用方的关联逻辑 |
| 只读不改 | 场景走读表格中所有步骤的"问题"列全为 `—`，但代码有变更 |
| 场景缺失 | 未达到 §5.3 最低场景目录要求 |
| 无平台差异 | 多平台产品中 `平台=all` 的场景未验证平台差异点 |
| 跳过异常路径 | 只有 happy path，无 error_path 或 edge_case 场景 |
| 问题无路径 | 记录了问题但无复现路径 |
| 引用无效行号 | 追溯矩阵中 ≥50% 的 `文件:行号` 引用指向不存在的代码行 |

**任一条件触发 → Professional Gate = Failed。**

## 8. 与现有评审体系的集成

### 8.1 与评审门控的关系（Gate Division）

场景追溯评审在 **Professional Gate** 执行时加载，作为 Professional Gate 的**核心检查项**。

场景追溯评审与 Professional Gate 其他检查项的分工：

| 维度 | 场景追溯评审 | Professional Gate 其他检查项 |
|------|------------|--------------------------|
| 核心问题 | "用户场景能不能跑通" | "代码质量是否合理" |
| 检查方式 | 逐场景走读代码 | 架构/安全/性能/可维护性检查 |
| 发现问题 | 场景不通、逻辑错误、功能缺失、场景不适配 | N+1 查询、过度抽象、命名不一致 |
| 裁决 | 场景不通 → Professional Gate Failed | 架构不合理 → Professional Gate Fail |

**场景追溯评审不是独立 gate**。所有场景追溯发现的裁决都在 Professional Gate 内完成，不直接触发 Value Gate 的失败。

**Value Gate 桥接规则**：当场景追溯评审发现 **P0 happy path 完全不通** 或 **功能缺失** 时，评审者必须在评审报告中注明"建议 Value Gate 重新评估"，并将发现的问题写入 `05-value-evidence.yaml` 作为 Value Gate 的补充证据。Value Gate 是否 re-fail 由 Value Gate 评审者独立裁决，场景追溯评审不代替 Value Gate 做裁决。

### 8.2 Gate 失败映射表

| 场景追溯发现 | 影响的 Gate | 裁决 |
|------------|------------|------|
| P0 happy path 不通 | Professional Gate（+ Value Gate 桥接建议） | Failed |
| P1 happy path 不通 | Professional Gate | Failed |
| 逻辑错误 (Critical) | Professional Gate | Failed |
| 功能缺失 | Professional Gate（+ Value Gate 桥接建议） | Failed |
| 接口不匹配 (Major) | Professional Gate | Failed（修复后可 Pass） |
| 场景不适配 (Major) | Professional Gate | Failed（修复后可 Pass） |
| 边界遗漏 (Major) | Professional Gate | Failed（需修复）或 Conditional Pass（评审者判断影响可控且有修复计划） |
| 体验降级 (Minor) | Professional Gate | Pass（≤3 个 Minor） |
| 资源泄漏 (Major) | Professional Gate | Failed（需修复）或 Conditional Pass（有修复计划） |
| 安全场景未覆盖 | Professional Gate | Failed |
| 场景目录不达标 | Professional Gate | Failed |
| 追溯矩阵缺失 | Professional Gate | Failed（无法评审） |

### 8.3 与 checklist 的关系

- `stage6-coding-review.md` 的"场景走读"检查项必须引用本规范的追溯矩阵
- `stage7-test-review.md` 的"场景追溯覆盖"检查项必须验证：追溯矩阵中每个场景路径都有对应测试用例
- 追溯矩阵是 Stage 6 和 Stage 7 评审的**共同输入**

### 8.4 加载时机

场景追溯评审在 **Professional Gate** 执行时加载。评审者必须先完成追溯矩阵，再执行场景走读。

## 9. 硬失败条件（汇总）

以下情况 Professional Gate 直接 Fail：

1. 任一 P0 功能没有对应场景路径走读
2. 场景走读发现 Critical 问题但未记录到问题表
3. 追溯矩阵中有场景路径但无对应代码位置
4. 触发 §7 任一浅层走读条件
5. 未达到 §5.3 最低场景目录要求
6. 追溯矩阵完全缺失
7. 触发 §6 任一真实性校验失败

## 10. 与需求文档的对标

### 10.1 PRD/需求 完整性

追溯矩阵的"需求来源引用"列必须逐条对标需求文档（PRD 或等效需求来源）。需求文档中每个功能条目必须在追溯矩阵中有对应场景。

### 10.2 范围偏差处理

| 偏差类型 | 处理方式 |
|---------|---------|
| PRD 要求但代码未实现 | 标注为 `功能缺失`（§5.2），按 Critical 处理 |
| 代码实现超出 PRD 范围 | 标注为 `需求来源=超范围`，在评审报告中注明"建议确认是否有意为之" |
| PRD 已过时（与当前需求不符） | 评审者按实际代码行为记录场景，在评审报告中注明"PRD 需要更新" |

### 10.3 隐式场景补充

需求文档未覆盖的隐式场景（如断网重连、首次启动、升级兼容、后台杀死）由评审者补充，标注为 `需求来源=隐式`。
