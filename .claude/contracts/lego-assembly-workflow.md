# 乐高拼装工作流契约（LEGO Assembly Workflow Contract）

## 1. 目的

定义任务开发从架构到代码块的多级拆装工作流，确保每个层级都有明确的设计、制造、拼装、评审和恢复机制。
将"从大积木到小积木"的拼装思维正式化为可执行的规范流程。

## 2. 适用范围

本契约在所有 Standard/Complex 任务的 build phase 中生效。
Trivial 任务可跳过多级拆装，单次 pass 完成。

## 3. 核心原则

1. **先设计后制造** — 任何积木在被制造前，必须有对应的设计稿说明它是什么、怎么拼、拼在哪
2. **多级拆装递归** — 成品 → 大结构件 → 结构件 → 组成单元 → 最小积木单元，每级拆解到可独立制造为止
3. **Agent 即工厂** — 每个子 Agent 是一个制造工厂，负责制造指定层级的积木单元
4. **拼装逐级向上** — 最小积木单元先拼装成功能组件，功能组件拼装成功能模块，功能模块拼装成完整软件
5. **异常可恢复** — 任何子 Agent 挂死、产出不足、质量不合格，必须有定义的回收和重新分配路径

## 4. 规则与决策模型

### 4.1 乐高层级模型（5 级）

| 层级 | 名称 | 对应物 | 设计文档 | 制造者 | 拼装者 |
|------|------|--------|---------|--------|--------|
| L0 | **成品级** | 完整软件/产品 | 方案详设（总体） | 全部下级拼装 | 主线程 |
| L1 | **模块级** | 功能模块（如"认证模块"、"订单模块"） | 方案详设（模块级） | L2 拼装结果 | L2 拼装结果 |
| L2 | **组件级** | 功能组件（如"用户列表组件"、"API 路由"） | 方案详设（组件级） | L3 拼装结果 | L3 拼装结果 |
| L3 | **单元级** | 最小功能单元（如"登录函数"、"查询接口"） | 方案详设（单元级） | L4 制造结果 | L4 制造结果 |
| L4 | **代码块级** | 最小积木单元（函数/类/组件/配置文件） | 方案详设（代码块级） | 子 Agent | L3 拼装 |

**规则**：
- 每个层级的积木必须定义：输入接口、输出接口、拼装方式、依赖的下一层级
- 同一层级内的积木之间应低耦合；不同层级间通过明确接口通信
- 拆解深度以"可独立制造"为停止条件：如果一个 L3 单元仍复杂到需要拆成 3+ 个独立代码块，则它应提升为 L2
- L4 代码块（函数/类/组件）必须能在 Design §14 中找到对应的 STEP 伪代码
- L4 代码块制造完成后，代码文件头部必须包含 `@packet` 注释块（见 §4.6.5）

#### 4.1.1 12 层 → L0-L4 映射表（P2-5 fix）

`work-packet-governance.md` §2 定义的 12 层拆包映射到乐高 5 级：

| 乐高层级 | work-packet-governance 层 | 说明 |
|---------|--------------------------|------|
| L0（成品） | — | 成品是全部 L1 模块拼装后的结果，不直接对应 12 层中的任何一层 |
| L1（模块） | 1. 需求边界、2. 数据结构 | 模块级拆包关注功能范围和数据 schema |
| L2（组件） | 3. 持久化对象、4. 数据访问、5. 请求契约、6. 响应契约 | 组件级拆包关注数据层和接口契约 |
| L3（单元） | 7. 服务接口、8. 业务逻辑、9. 控制器入口 | 单元级拆包关注业务实现和入口编排 |
| L4（代码块） | 10. 集成接线、11. 验证回归、12. 可运维性 | 代码块级拆包关注配置、测试、可观测性 |

**映射规则**：
- 若 12 层中的某层被标记为 `not_applicable`，其对应的乐高层级仍必须在拆解树中存在（可为空节点），以保持层级结构完整性
- 8a. 函数级拆包属于 L4，当单个业务逻辑规则需要独立制造时，在 L4 中创建独立代码块

### 4.2 7 阶段 × 乐高层级映射

| 7 阶段 | 主要涉及的乐高层级 | 产出 | 评审焦点 |
|--------|-------------------|------|---------|
| 1 深度学习 | L0 + L1 | 学习分析报告：竞品架构拆解、技术选型对比 | 是否覆盖 ≥3 参考项目 + ≥2 技术方案 |
| 2 架构拆解 | L0 → L4 | 架构拆解清单：功能树（成品→模块→组件→单元→代码块） | 是否拆到最小可独立制造单元 |
| 3 PRD | L0 + L1 | PRD 文档：产品级需求、用户场景、功能边界 | 是否回链到用户真实诉求 |
| 4 方案详设 | L1 → L4 | 详细设计：每个模块/组件/单元的接口定义、数据流、数据库 schema、通信协议 | 是否有足够细节让子 Agent 直接制造 |
| 5 开发计划 | L1 → L4 | 开发计划：每个代码块对应一个 work packet，定义 Agent 分配、依赖、里程碑 | 是否每个 packet 有明确 objective + acceptance + verification |
| 6 编码 | L4 | 代码：每个子 Agent 制造一个代码块（函数/类/组件/配置） | 安全 + 正确 + 性能 + 风格 |
| 7 测试 | L3 → L0 | 测试报告：单元→组件→模块→成品，逐级验证拼装正确性 | 覆盖率 + 测试类型 + 可执行性 |

### 4.3 深度学习目标定义

深度学习（Stage 1）必须产出以下内容，按从大到小的顺序：

1. **架构级**：参考项目的整体架构是什么样的？用了什么框架/协议/部署方式？
2. **组件级**：核心组件有哪些？组件间怎么通信？同步还是异步？
3. **接口级**：对外暴露什么 API？参数格式？错误码？
4. **数据级**：用什么数据库？表结构？索引策略？
5. **实现级**：关键函数/类的设计模式？已知坑点？
6. **竞品对比**：至少 3 个参考项目的优缺点对比表

输出要求：《学习分析报告》≥2000 字，包含竞品对比表、技术选型依据、已知坑点清单。

### 4.4 PRD 文档深度定义

PRD（Stage 3）必须从大到小覆盖：

1. **产品全貌**：这个软件是什么？解决什么问题？目标用户？
2. **功能架构**：有哪些功能模块？模块间关系？优先级？
3. **用户体验**：界面布局？交互流程？切换与动画？视觉风格？
4. **数据模型**：核心实体？实体关系？数据生命周期？
5. **接口定义**：对外 API？对内接口？第三方集成？
6. **约束条件**：性能要求？安全要求？合规要求？

PRD 是"产品级设计稿"，回答"做什么"和"为什么做"，不回答"怎么做"。

### 4.5 方案详设深度定义

方案详设（Stage 4）是"制造级设计稿"，回答"怎么做"。对于 `depth_profile=implementation_blueprint` 的文档，必须包含以下强制章节：

1. **总体架构**：系统分层、部署拓扑、技术栈选型依据
2. **模块设计**：每个模块的：
   - 职责边界（做什么 / 不做什么）
   - 输入 / 输出接口
   - 内部子模块划分
   - 数据流（同步 / 异步）
   - 错误处理与降级策略
3. **组件设计**：每个组件的：
   - 在模块中的位置
   - 接口定义（方法签名 / 参数 / 返回值）
   - 状态管理
   - 与其他组件的依赖关系
4. **单元设计**：每个单元的：
   - 算法/逻辑描述
   - 输入校验
   - 输出格式
   - 异常处理
5. **数据库设计**：表结构 / 索引 / 约束 / 迁移
6. **通信设计**：UI ↔ Server ↔ DB 的：
   - 协议选择（HTTP/gRPC/WebSocket/...）
   - 数据格式（JSON/Protobuf/...）
   - 认证 / 授权
   - 重试 / 超时 / 降级

#### 4.5.1 强制章节 §14-§18（L4 函数级蓝图）

对于 `depth_profile=implementation_blueprint` 的详设文档，必须在 §6-§8 基础上补充以下 5 个强制章节：

**§14 函数级调用链设计**
- §14.0 跨模块依赖注入总表（模块 | 必需依赖 | 注入来源）
- §14.N 流程 N：STEP 1→STEP M 逐函数调用链伪代码
- 每条流程末尾必须包含：异常传播链、状态变更追踪
- 伪代码粒度达到 if/try/for/await 级别

**§15 函数复用矩阵**
- 函数签名 | 定义模块 | 被调用模块 | 调用次数 | 复用类型

**§16 接口复用声明**
- 接口名 | 定义位置 | 消费者 | 消费方式（注入/直接引用/事件订阅）

**§17 通信协议逐消息设计**
- 消息类型 | 方向 | 字段列表 | 序列化 | 超时 | 重试 | 降级策略

**§18 单元评审与合入计划**
- 单元 ID | 评审人 | 合入前置条件 | 合入目标分支 | 回滚策略

**硬规则**：方案详设的粒度必须达到"AI Agent 拿到设计稿后，不需要额外决策就能写出代码，且每行代码的来源可追溯到 §14 的某一行伪代码"。

### 4.6 Agent 集群架构

#### 4.6.1 顺序拼装模型

```
主线程（Orchestrator）
  ├── 拼装 L1 模块 A → 派发 Agent-A（负责模块 A 的拼装）
  │     ├── 拼装 L2 组件 A1 → 派发 Agent-A1（负责组件 A1）
  │     │     ├── 制造 L3 单元 A1a → 派发 Agent-A1a
  │     │     │     └── 制造 L4 代码块 A1a-1 → 直接 Write/Edit
  │     │     └── 制造 L3 单元 A1b → 派发 Agent-A1b
  │     └── 拼装 L2 组件 A2 → 派发 Agent-A2
  └── 拼装 L1 模块 B → 派发 Agent-B（负责模块 B 的拼装）
        └── ...
```

**规则**：
- 主线程只负责 L0 级别的拼装（将所有 L1 模块拼装成成品）
- 每个 Agent 负责自己层级内的拼装，向下派发制造任务
- 拼装采用**顺序逐层 + 层内 Fan-Out**模式（非递归嵌套）：主线程 Fan-Out L4 制造 → Fan-In → 拼装为 L3 → 验证 → Fan-Out L3 → ... → L0
- 同一层级内的拼装可以并行
- 子 Agent 不得再嵌套派发子 Agent（避免嵌套过深；所有 Fan-Out 由主线程统一调度）

#### 4.6.2 Agent 分配矩阵

每个 work packet 对应一个 Agent 任务，必须包含：

```yaml
packet_id: <id>
lego_level: L1|L2|L3|L4          # 当前负责的乐高层级
objective: "制造/拼装 <具体名称>"  # 单一、明确、可判定完成的目标
timeout_minutes: <N>               # 预计超时阈值（挂死检测 = 2× 此值）
read_scope:                       # 可读的设计文档范围
  - "方案详设 §X.X"
  - "相关依赖模块的接口定义"
write_scope:                      # 可写的代码范围
  - "src/module_x/component_y/"
  - "tests/module_x/component_y/"
dependencies: [<依赖的 packet_id>] # 必须先完成的其他 packet
acceptance:                       # 怎么算完成
  - "函数 X 通过所有单元测试"
  - "组件 Y 渲染正确且响应 <100ms"
verification:                     # 怎么验证
  - "运行 npm test -- component_y"
  - "lint 无 error"
rollback:                         # 如何回滚
  - "git revert <commit>"
```

**timeout_minutes 默认范围**（未在开发计划中显式指定时生效）：

| 乐高层级 | 默认 timeout_minutes | 挂死检测阈值（2×） |
|---------|---------------------|-------------------|
| L4（代码块） | 5 | 10 min |
| L3（单元） | 15 | 30 min |
| L2（组件） | 30 | 60 min |
| L1（模块） | 60 | 120 min |

开发计划（Stage 5）可根据具体任务复杂度覆盖默认值，但不得低于层级最小值的 50%。

#### 4.6.3 Fan-Out / Fan-In 工作流

多 Agent 执行必须遵守 Fan-Out → Wait → Fan-In：

1. **Plan**：主线程分析当前乐高层级，识别可并行的制造任务
2. **Fan-Out**：一次性拉起所有子 Agent（同一消息中发出），避免边做边补
3. **Wait**：等待子线程收敛；禁止高频轮询
4. **Fan-In**：主线程回收结果，检查：
   - 所有 packet 是否完成
   - 产出是否符合 acceptance 标准
   - 模块间拼装是否正确（接口对齐、数据类型一致）
5. **Synthesize**：主线程形成统一结论，拼装到上一级

#### 4.6.4 Agent 异常恢复

| 异常类型 | 检测方式 | 恢复动作 |
|---------|---------|---------|
| Agent 挂死（超时） | 超过 `timeout_minutes × 2` 未返回 | 终止 → 重新派发（附原始 spec + 失败分析）→ 记录 exception |
| Agent 产出不足 | Fan-In 时发现 acceptance 未满足 | 标记 incomplete → 重新派发补全（附缺失项清单） |
| Agent 产出质量不合格 | 评审 Fail | 标记 failed → 修复 → 复评（最多 3 轮） |
| Agent 依赖未就绪 | 依赖的 packet 未完成 | 放入等待队列 → 依赖完成后重新派发 |
| 上下文预算不足 | 主线程 context → compaction | 执行 compaction → checkpoint → 继续 |
| **Agent 产出部分正确但不完整** | Fan-In 时发现产出包含正确部分但覆盖不全 | 提取正确部分 → 标记缺失项 → 重新派发补全（附已产出内容作为上下文） |
| **多 Agent 输出冲突** | Fan-In 时发现多个产出修改同一文件/接口 | 标记冲突文件 → 主线程手动裁决 → 记录 resolution → 重新派发冲突方 |
| **Agent 执行中崩溃** | Agent tool 返回错误或 session 断开 | 记录崩溃点 → 重新派发（附原始 spec + 已完成的 work product） |
| **主线程上下文过期** | 主线程 compaction 后无法恢复拼装链状态 | 从最近 checkpoint 恢复 → 重建拼装链 → 验证已拼装层级 |

**恢复优先级**：主线程上下文过期 > 挂死 > 崩溃 > 产出部分正确 > 产出不足 > 质量不合格 > 依赖未就绪 > 输出冲突

**重新派发上下文规则**：
- 首次派发：原始 packet spec（objective + read_scope + write_scope + acceptance）
- 重新派发：原始 spec + 失败产出（如有） + 失败原因分析 + 修正指引
- 质量不合格复评：原始 spec + 评审报告 + 修复计划

**质量不合格修复归属**：优先由原 Agent 修复（有上下文）；原 Agent 不可用时由新 Agent 接手（需传递完整上下文）；复杂冲突由主线程直接修复。

#### 4.6.5 L4 代码块头部注释模板

每个 L4 代码块制造完成后，代码文件头部必须包含 `@packet` 注释块。注释模板遵循 `execution-traceability` §4.2 的标准定义：

```typescript
/**
 * @packet {packet_id}
 * @input  {从哪个包读什么数据}
 * @output {对外导出什么}
 * @contract {必须满足的接口签名}
 * @ref    {Design §14.x STEP N — 精确伪代码位置}
 * @test   {验证一句话}
 * @oversized {true|false}
 * @oversized_reason {if true: 纯数据结构|原子逻辑|同模式批量|强耦合胶水}
 */
```

**硬规则**：无 `@packet` 注释的代码文件不得进入拼装阶段。

#### 4.6.6 L4 派发判定标准

L4 代码块（函数/类/组件/配置）的制造由以下规则决定派发方式：

| 判定条件 | 派发方式 |
|---------|---------|
| 预估代码量 < 50 行 **或** 预期工具调用 ≤ 3 次 | 主线程直接 Write/Edit |
| 预估代码量 ≥ 50 行 **或** 需要独立调研/测试 | 派发子 Agent |
| 涉及安全敏感代码（认证、加密、权限） | 必须派发子 Agent（独立评审） |
| 与其他 L4 代码块有强依赖 | 优先主线程（减少上下文切换成本） |
| 批量同类代码块（≥3 个同模式函数/组件） | 派发子 Agent（批量产出） |

**规则**：主线程在 Fan-Out 前必须对每个 L4 packet 应用此判定，不满足子 Agent 派发条件的直接用主线程制造。

### 4.7 拼装正确性验证

每级拼装完成后必须验证：

| 拼装层级 | 验证内容 | 验证方式 |
|---------|---------|---------|
| L4 完成 | 代码块头部 @packet 注释、@contract 对齐、单元测试 | 包级即时评审 |
| L4 → L3 | 代码块是否正确实现单元逻辑 | 单元测试 + 拼装级评审 |
| L3 → L2 | 单元是否正确组装成组件 | 集成测试 + 拼装级评审 |
| L2 → L1 | 组件是否正确组装成模块 | 模块测试 + 拼装级评审 |
| L1 → L0 | 模块是否正确组装成成品 | 端到端测试 + 拼装级评审 |

**硬规则**：
- 低层级拼装未通过验证前，不得进行高层级拼装
- 拼装失败时，回退到失败的层级重新拼装，不是重做全部

### 4.8 评审与审计

每个乐高层级的产出都必须通过评审：

| 层级 | 评审时机 | 评审方式 | 评审焦点 |
|------|---------|---------|---------|
| L0（成品） | Stage 7 测试通过后 | closeout 评审 | 用户价值 + 完整交付 |
| L1（模块） | 模块拼装完成后 | 独立评审 Agent | 模块边界 + 接口正确性 |
| L2（组件） | 组件拼装完成后 | 独立评审 Agent | 组件职责 + 状态管理 |
| L3（单元） | 单元实现完成后 | 代码评审 | 逻辑正确 + 错误处理 |
| **L4（代码块）** | **代码写完后** | **包级即时评审** | **@contract 对齐 + §14 伪代码一致 + 单元测试** |

评审 verdict 映射：
- **Pass** → 进入下一级拼装
- **Conditional Pass** → 记录修复计划，允许进入下一级拼装但必须在本级 closeout 前修复
- **Fail** → 修复后复评，最多 3 轮

## 5. 输出与投影

本契约产出的正式文件：
- `tasks/<task-id>/artifacts/lego-decomposition-tree.yaml` — 多级拆解树
- `tasks/<task-id>/artifacts/agent-assignment-matrix.yaml` — Agent 分配矩阵
- `tasks/<task-id>/reviews/` — 各级拼装评审结果
- `tasks/<task-id>/exceptions/` — Agent 异常恢复记录

## 6. 边界

### 6.1 与其他契约的边界

- 与 cluster-orchestration：本契约定义"怎么按乐高层级拆装"，cluster-orchestration 定义"多 Agent 怎么编排"
- 与 work-packet-governance：本契约定义 work packet 的乐高层级归属，work-packet-governance 定义 packet 的拆包规则
- 与 review-gates：本契约定义各级评审时机，review-gates 定义评审的门控规则

### 6.2 不覆盖的范围

本契约不定义：
- 具体代码的写法（由方案详设和工程标准定义）
- 评审的具体 checklist（由各 Stage checklist 定义）
- 工具的具体调用（由 skill-tool-mapping 定义）

## 7. 异常边界

以下场景允许偏离本契约：
- Trivial 任务：跳过多级拆装，单次 pass 完成
- 用户明确要求快速迭代：减少评审层级，但风险记录到 exception
- 工具/Agent 不可用：走 fallback 或 exception

以下场景**永远不允许**偏离：
- 未设计就制造（没有方案详设就写代码）
- 拼装不验证就向上拼装（低层级验证未通过就进行高层级拼装）
- Agent 异常不恢复就宣称完成

## 8. 验证要求

| checker_id | 检查内容 | 失败后果 |
|-----------|---------|---------|
| `decomposition-tree-complete` | 是否从 L0 拆到 L4 且有明确停止条件 | professional gate failed |
| `agent-assignment-matrix` | 每个 work packet 是否有明确的目标/范围/验收 | contract gate failed |
| `assembly-chain-valid` | 拼装链是否从 L4→L0 逐级验证 | professional gate failed |
| `recovery-plan-defined` | 是否有 Agent 异常恢复计划 | contract gate failed |

## 9. 迁移策略

从"整体开发"到"多级乐高拆装"的过渡：

1. **第一阶段**：引入乐高层级概念，任务开始时必须定义 L0-L4 的拆解树
2. **第二阶段**：开发计划（Stage 5）必须包含 Agent 分配矩阵，每个 packet 对应一个代码块
3. **第三阶段**：编码（Stage 6）按乐高层级逐级拼装，每级拼装后验证
4. **第四阶段**：Agent 异常恢复自动化，挂死/产出不足自动检测并重新派发

## 10. 非目标

本契约不追求：
- 无限级拆解 — 5 级（L0-L4）是经验上限，所有 Fan-Out 由主线程统一调度，子 Agent 不再嵌套
- 每个代码块都必须由 Agent 制造 — L4 级代码块可由主线程直接 Write/Edit
- 取代方案设计 — 乐高拼装是方案的执行面，方案详设仍然是设计的真相源
- 消灭所有人工决策 — 关键架构决策仍需人类判断，Agent 负责执行
