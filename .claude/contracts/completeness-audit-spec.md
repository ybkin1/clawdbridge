# 完整性审计规范（Completeness Audit Specification）

## 1. 问题定义

这是本规范架构中最关键的自我约束规则。它解决的问题：

**Agent 在执行任务时，倾向于对源材料做摘要式压缩而非完整性映射，导致交付物只有骨架没有实质内容。**

具体表现：
- 读了 500 行源材料，只写 50 行摘要
- 列出了"有这个东西"但没有展开结构
- 用"见原版"替代实际内容
- 把应独立成文的规范合并成一段
- 交付后等用户发现缺失才补充

## 2. 根因

现有规则只约束"怎么做"（Karpathy 原则、多Agent、复杂度分级），不约束"做完后是否完整"。

没有以下三道防线：
1. 完整性对标：源材料有什么，交付物必须逐项确认覆盖或显式声明不覆盖
2. 交付物成熟度检查：文档是否达到该类型的默认深度
3. Self-audit：交付前必须自检"我是否偷懒压缩了内容"

## 3. 三道防线规则

### 防线 1：完整性对标（Completeness Mapping）

每次从源材料（Codex 原版、外部文档、参考资料）提取内容到交付物时，**必须在交付物末尾附完整性对标表**：

```markdown
## 完整性对标

| 源材料条目 | 是否已适配 | 适配位置 | 未适配原因 |
|-----------|----------|---------|-----------|
| Codex 原版 §4 Task Truth Layer Objects | ✅ 已适配 | task-tracking-contract.md §2, §7 | - |
| Codex 原版 §6.1 Required Fields for state | ⚠️ 部分适配 | task-tracking-contract.md §4 | 字段由 state-contract.yaml 持有，本文不再重复 |
| Codex 原版 §10 Parallel Collaboration | ❌ 未适配 | - | 待补充到 task-tracking-workflow-spec.md |
```

**规则**：
- 每条源材料条目必须明确标记 ✅ / ⚠️ / ❌
- ⚠️ 必须说明哪些部分未适配
- ❌ 必须说明原因 + 补充计划
- 不得用"已覆盖"含糊带过，必须指出具体适配位置

### 防线 2：交付物成熟度检查（Maturity Gate）

每次交付文档类产出前，必须检查是否达到该类型的默认深度。成熟度等级体系详见 `document-depth.md` §4，以下阈值是最低要求：

| 文档类型 | 最低字数 | 最低检查项 |
|---------|---------|-----------|
| 契约/规范 | ≥3000字 或 源材料 60%（取较高值） | 目的/原则/结构/规则/边界/异常/迁移/验收 |
| 学习报告 | ≥2000字 或 源材料 40%（取较高值） | 背景/现状/对比/结论/残余问题 |
| 架构拆解 | ≥1500字 或 源材料 30%（取较高值） | 功能树/选型/交互/维护 |
| PRD | ≥3000字 或 源材料 50%（取较高值） | 目标/用户/场景/功能/验收/非目标/风险 |
| 评审报告 | ≥1000字 | 检查面/finding/结论/建议 |
| 流程/规范 | ≥2000字 或 源材料 50%（取较高值） | 触发条件/步骤/角色/输入输出/异常/恢复 |

注：对于小型任务（单一 API 适配、配置变更等轻量 Standard 任务），上述绝对字数阈值和百分比阈值**取较低值**（即：两者中较低的那个即为最低要求），但交付物必须包含完整的实质性内容（不能只有标题和骨架）。"轻量 Standard" 由主线程在复杂度判定阶段标注到 `00-task-state.yaml` 的 `delivery_mode = quick` 字段中。

**硬规则**：
- 低于最低字数时，不得宣称 `contract-ready` 或 `implementation-ready`
- 必须在交付物开头声明当前成熟度（见 `document-depth.md` §4）：`maturity: authoring-ready | contract-ready | draft`
- 若源材料 500 行但交付物 <50 行，自动判定为 `summary` 深度，不得伪装成 `detailed`

### 防线 3：Self-Audit（交付前自检）

每次任务交付前（向用户报告"完成"之前），必须执行以下自检：

```markdown
## Self-Audit

### 完整性
- [ ] 我是否逐条对比了源材料和交付物？
- [ ] 源材料中有哪些重要段落我没有适配？为什么？
- [ ] 我是否把应该独立成文的内容合并成了摘要？

### 成熟度
- [ ] 交付物是否达到该类型的默认深度？
- [ ] 如果源材料 500+ 行，我的交付物是否有实质内容而非骨架？
- [ ] 我是否写了"详见XX"来替代实际内容？

### 执行质量
- [ ] 我是否跳过了该走的阶段（如架构拆解）直接写产出？
- [ ] 我是否先写产物再事后解释"这次做了什么"？
- [ ] 我是否对用户的"还有别的吗"类问题做了敷衍回答？
```

**硬规则**：
- 任何自检项回答为"是"（负面项）时，必须在交付给用户之前修复
- 不得跳过自检直接报告完成
- Self-audit 结果必须写入交付物或聊天输出

## 4. 与文档深度契约的关系

本规范是 document-depth 契约的**执行面**：
- document-depth 定义了"该有多深"
- completeness-audit 定义了"怎么确保达到了这个深度"

## 5. 触发条件

以下情况自动触发完整性审计：
1. 从源材料提取/适配内容（任何 Standard/Complex 任务）
2. 用户追问"还有别的吗"/"是否完整"/"有没有遗漏"
3. 评审 Agent 发现交付物明显薄于源材料
4. 用户指出"你写得太薄了"/"这跟我给的材料不匹配"

## 6. 违反处理

违反完整性审计规则的后果：

| 违反类型 | 处理 |
|---------|------|
| 未做完整性对标就交付 | 退回重写，补充对标表 |
| 成熟度不达标却宣称 ready | 降级为 draft，重新评审 |
| Self-audit 发现偷懒但未修复 | blocked，报告用户 |
| 连续 2 次完整性问题 | 强制进入 Complex 评审模式（3-4 Agent） |

## 7. 与现有规则的关系

本规范不替代现有规则，而是在现有 5 条元规则之上增加**完整性防线**：

```
Rule 1 (前置学习) → 确保读了
Rule 2 (评审循环) → 确保评审了
Rule 3 (意图确认) → 确保理解了
Rule 4 (脏数据清理) → 确保干净了
Rule 5 (非C盘存储) → 确保位置对了

→ 新增 Rule 6 (完整性审计) → 确保完整了，没偷懒
```
