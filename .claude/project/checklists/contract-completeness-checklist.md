# 规范完整性检查清单（Contract Completeness Checklist）

> 当任务涉及修改 `.claude/contracts/` 目录下的契约文件，或新增规范类交付物时，**必须使用本清单**进行评审。
> 本清单用于补充现有各 Stage checklist 中缺失的"外部对标"维度。

## 1. 注册表完整性

- [ ] `registry.yaml` 中所有 `status=active/provisional` 的契约，是否都有对应的 `.md` 文件？
- [ ] 每个契约文件是否都有 YAML front matter？
- [ ] front matter 的 `contract_id` 是否与 registry.yaml 中的 entry key 一致？
- [ ] 是否有 registry 中有条目但文件不存在的情况？
- [ ] 是否有文件存在但 registry 中没有条目的情况？

## 2. 契约结构合规性

对每个被修改/新增的契约，检查：

- [ ] 是否包含 contract-schema.md 定义的 13 个 metadata 字段？
- [ ] 是否包含 contract-schema.md 要求的 10 个语义章节？（目的/适用范围/核心原则/规则决策/输出投影/边界/异常边界/验证要求/迁移策略/非目标）
- [ ] Spec 类文件（`-spec.md`）是否包含最小 4 章节？（目的/规则/触发条件/违反处理）

## 3. 契约间依赖闭合

- [ ] 契约 front matter 中的 `required_contracts` 是否与 registry.yaml 中的 `depends_on` 一致？
- [ ] 契约正文中引用的其他契约是否存在且 active？
- [ ] `conflicts_with` 是否已填写？（不得留空而不填写，如无冲突应填 `[]`）

## 4. 源材料完整性对标（仅当任务涉及从外部源材料提取内容时）

- [ ] 交付物是否附有完整性对标表，逐条标注源材料条目？
- [ ] ✅ 适配的条目是否指出了具体适配位置（文件+章节）？
- [ ] ⚠️ 部分适配的条目是否说明了哪些部分未适配？
- [ ] ❌ 未适配的条目是否说明了原因 + 补充计划？
- [ ] 是否有源材料中的重要条目被完全遗漏？
- [ ] 是否有把应该独立成文的内容合并成了摘要？

## 5. 规范执行面验证

- [ ] 被修改的契约中声明的规则，是否有对应的执行机制？（不能只声明规则，没有怎么检查/怎么执行）
- [ ] 规则的违反处理是否明确？
- [ ] 规则是否可验证？（不可验证的规则视为未定义）

## 6. 交叉引用一致性

- [ ] 契约正文中引用的文件路径是否存在？
- [ ] 契约中引用的章节号是否与实际章节一致？
- [ ] 多个契约之间对同一概念的定义是否一致？（如 phase 状态机、路由词汇等）

## 7. 变更影响面

- [ ] 本次修改影响了哪些契约？
- [ ] 被影响的契约的 `depends_on` 列表是否需要同步更新？
- [ ] 本次修改是否引入了与其他契约的冲突？
- [ ] 如果修改了 registry.yaml，是否同步更新了 CLAUDE.md Part E 的架构总览？

## 8. AI 可执行性验证（新增）

> 本维度防止"结构合规但 AI Agent 无法直接执行"的盲区。 Round 3 深度压力测试发现的问题应在本维度被提前拦截。

### 8.1 Checker 执行粒度

- [ ] 每个 `mode=manual` 的 checker，是否包含 ≥5 步的编号执行步骤？（步骤必须具体到 AI Agent 能按序执行，不能只有"检查面"概述）
- [ ] 每个 `mode=hybrid/automated` 的 checker，是否声明了**语言/技术栈 scope**？（如只适用于 JS/TS，或适用于所有 typed-language）
- [ ] 每个 checker 是否声明了**无支持环境时的降级策略**？（如纯 JS 项目无类型检查时的 fallback）
- [ ] 算法描述中是否不存在自然语言模糊词？（禁止："适当处理"、"必要时"、"视情况而定"、"合理范围"；必须替换为可判定条件）

### 8.2 产出格式可解析性

- [ ] 契约中定义的所有产出（report、receipt、matrix、exception 记录），是否附有**机器可解析的 schema 示例**？（至少一个完整 YAML/JSON 示例）
- [ ] schema 示例中是否包含所有必填字段和取值枚举？
- [ ] 若产出用于阻断流程（如阻断拼装、阻断阶段切换），是否明确定义了**阻断条件判定规则**？（不能只有"产出后由人工判断"）

### 8.3 自检与运行时闭合

- [ ] 若契约要求"主线程自检"或"持续审计"，是否提供了**自检清单**？（具体检查项 + 通过标准 + 失败处理）
- [ ] 自检机制是否声明了**触发时机**和**最大耗时目标**？
- [ ] 自检产出的 warning 是否有明确的**清零规则**？（何时必须解决、何时可记录为 exception）
