# 完整性审计评审清单

> 本清单是 document-depth 契约的执行面，定义"怎么确保交付物达到了要求的深度"。
> **规范类交付物**（修改 `.claude/contracts/` 下的契约文件）**必须同时加载** `contract-completeness-checklist.md`。

## 完整性对标检查

- [ ] 交付物是否附有完整性对标表？
- [ ] 对标表是否逐条标注了源材料条目？
- [ ] ✅ 条目是否指出了具体的适配位置（文件+章节）？
- [ ] ⚠️ 条目是否说明了哪些部分未适配及原因？
- [ ] ❌ 条目是否说明了原因 + 补充计划？
- [ ] 是否有用"已覆盖"含糊带过的情况？

## 成熟度检查

- [ ] 契约/规范类：≥3000字 或 ≥源材料60%？
- [ ] 流程/规范类：≥2000字？
- [ ] 学习报告：≥2000字？
- [ ] 评审报告：≥1000字？
- [ ] 是否声明了当前成熟度（draft / authoring-ready / contract-ready）？
- [ ] 如果源材料 500+ 行，交付物是否有实质内容而非骨架？

## Self-Audit 检查

- [ ] 是否逐条对比了源材料和交付物？
- [ ] 源材料中是否有重要段落没有适配？为什么？
- [ ] 是否把应该独立成文的内容合并成了摘要？
- [ ] 是否跳过了该走的阶段（如架构拆解）直接写产出？
- [ ] 是否先写产物再事后解释"这次做了什么"？
- [ ] 是否对用户的"还有别的吗"类问题做了敷衍回答？

## 结构完整性检查

- [ ] 契约是否覆盖：目的/原则/结构/规则/边界/异常/迁移/验收？
- [ ] 流程/规范是否覆盖：触发条件/步骤/角色/输入输出/异常/恢复？
- [ ] 学习报告是否覆盖：背景/现状/对比/结论/残余问题？
- [ ] 架构拆解是否覆盖：功能树/选型/交互/维护？
- [ ] 是否有"详见XX"来替代实际内容？
